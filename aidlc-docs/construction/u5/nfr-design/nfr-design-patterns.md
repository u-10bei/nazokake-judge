# U5 NFR Design Patterns — 出題停止（item retirement）

U5-NFR-01〜13 を設計パターン **DP-U5-NN** に落とす。U5 の設計は新奇な部品の導入ではなく、**既存部品の分割と集約**に尽きる。共通する狙いは **「新規のみ反映」「それまでの結果は有効」という要件の両輪を、規約ではなく構造で守る**こと。NFR Design 全 5 問 A（2026-07-17）。

---

## DP-U5-01: 読み取り経路の分割を「関数名」で強制する（最重要）

要件の両輪は「**active のみを読む経路**」と「**全件を読む経路**」の二本立てで初めて成立する（BR-U5-02）。これを**関数の分割**で固定する:

- 🔒 **`list_items()` は凍結**（SELECT 列・WHERE ともに一切変更しない・全件を返し続ける）
- 🆕 **`list_active_items()` を新設**（`WHERE retired_at IS NULL`）。**新規セッション系と充足判定のみが呼ぶ**

**なぜ引数フラグ（`list_items(active_only=True)`）にしないか**: 既定値の反転・呼び出し漏れという**ありふれた変更**で、要件の**両輪が同時に壊れる**:

| 壊れるもの | 機序 |
|---|---|
| **それまでの結果は有効** | export の `items` が縮む → 自己完結性（judgments の item ⊆ items）が破れる → **PU3-3 違反 → U4b 破壊**（judgments には出るが items にない item を assemble が黙って落とす） |
| **新規のみ反映** | 旧セッションのフォールバック導出が変わる → **進行中セッションの Likert ターゲットが変わる** |

**関数名で用途を分離**すれば、劣化経路は「凍結された関数を書き換える」という**明白な違反**としてしか到達できない。

- **判別の原理**: **「これから出題するものを選ぶ」なら active / 「既に起きたこと・既に配ったものを解決する」なら全件。**
- **検証（両輪それぞれに独立した網）**: **PU5-2**（旧セッションの Likert が廃止前後で一致）と **PU5-4**（廃止前後で export の `items` 集合が不変）。**劣化実装は両方で落ちる**。
- 対応: U5-NFR-04/05, BR-U5-02/02a/02b, Q2。

## DP-U5-02: 導出状態の保存化 + 単一アクセサ（原子性は既存 batch に相乗り）

**問題**: ペア列は開始時に**保存**されるのに、Likert ターゲットだけ**毎回導出**される非対称。プールを絞ると進行中セッションのターゲットが変わる。

**解決の 3 点セット**:

1. **保存化**: `Session.likert_targets: list[str] | None` をモデルに載せ、開始時に確定・保存する。**`exposure_snapshot`（JSON カラム ↔ 型付きフィールド）の前例に完全に従う**（読み書きのイディオムを増やさない）。
2. **原子性を既存 batch に相乗りさせる（★本命）**: **`save_pair_sequence` の同一 batch で INSERT** する。同関数は既に **Session + PairSequence を all-or-nothing 保存**する設計（TSD-03「半端なペア列を原理的に生じさせない」）。同じ batch に載せれば「**ペア列は保存されたが Likert ターゲットは未保存**」という中間状態が**原理的に生じない**。別経路で保存すると、その窓を**新設してしまう**（＝BR-U5-04 の保証に穴）。
3. **単一アクセサに集約**: `get_likert_targets` 1 本に統一する。現状 `select_likert_targets` の導出は **3 箇所に散在**（`build_view`=表示 / `check_complete`=完了判定 / `submit_likert`=検証）。**一部だけ保存値に切り替えると「表示されたターゲットを送信すると拒否される」**（表示=保存値・検証=導出値のずれ）。

- **副次効果**: PBT-02（U5-NFR-07 ラウンドトリップ）が `Session` モデル経由で自然に載り、**XC-02（セッション状態のラウンドトリップ）の対象に自然に入る**。
- **本要件がなくても正しい形**: 導出と保存の非対称は既存設計の歪みであり、U5 はそれを是正するだけ。
- 対応: U5-NFR-03/07, BR-U5-04, Q1/Q3。

## DP-U5-03: 冪等性を SQL の WHERE 句で作る

冪等性を**アプリ側の分岐に頼らず SQL に埋める**:

- 廃止: `UPDATE items SET retired_at = ? WHERE item_id IN (...) AND retired_at IS NULL`
  → **既に廃止済みは自然に no-op**＝**初回の廃止時刻が保持される**（BR-U5-06 を SQL 自体が保証）。証跡が後から書き換わらない。
- 復活: `UPDATE items SET retired_at = NULL WHERE item_id IN (...) AND retired_at IS NOT NULL`

**分類は報告用と割り切る**: `retired` / `already_retired` / `not_found` は **UPDATE 直前の SELECT** で判定する（batch 直前・ロックなし＝U4a 凍結ガードと同じ窓最小化方針）。**競合で分類表示が僅かにずれても実害はない**——冪等性は WHERE 句が保証しており、分類は運用者への報告にすぎない。管理 API は運用者の逐次操作でありロックは過剰。

- **全パラメータ化**: `IN (...)` は件数分の `?` をバインド（U5-NFR-12）。
- **検証**: PU5-3（複数回適用で状態同一・再廃止で初回時刻保持）。
- 対応: U5-NFR-11/12, BR-U5-06/07, Q4。

## DP-U5-04: 経路分離によるガードの保全

**凍結ガード（BR-U4a-03）と廃止を、同じ関数に混ぜない**:

- `insert_items` の参照済みガードは **無改修**（`body`/`layer`/`body_ref` の UPDATE は引き続き拒否＝プール凍結は維持）。
- 廃止は **別関数（`retire_items`）・別ルート（`/admin/items/retire`）** を通る。→ **参照済み item でも廃止できる**（BR-U5-05: `retired_at` は body/layer を変えず、過去判定の解釈を壊さないためガードの趣旨に反しない）。
- **型で穴を塞ぐ**: **`Item` 契約に `retired_at` を持たせない**（BR-U5-12）。→ `pool_ingest` のペイロードに `retired_at` が現れない＝**投入経路から廃止・復活が構造的に不可能**。ガードを緩めるのではなく、**対象外の経路を型で隔離**する。
- **検証（unit）**: 参照済み item の廃止が成功する／`pool_ingest` 再投入で `retired_at` 不変（BR-U5-08）。
- 対応: U5-NFR-08, BR-U5-05/08/12, Q4 前提。

## 導入しない設計部品（意図的な非採用・U1〜U4b と同方針）

| 部品 | 非採用理由 |
|---|---|
| キャッシュ | プール約 95 件・D1 の単純クエリ。 |
| キュー / 非同期 | UPDATE 数件・運用者の逐次操作。 |
| サーキットブレーカ / リトライ | 外部依存なし（既存 D1 のみ）。 |
| 分散ロック | 管理 API の逐次操作。冪等性は SQL の WHERE 句が保証（DP-U5-03）ゆえロック不要。 |
| **`retired_at` のインデックス** | プール約 95 件・全走査で十分（FD domain-entities §1）。 |
| 新規ライブラリ / 新規サービス | 標準 `json` / `urllib` のみで完結。 |
| スケール対策 | 規模・性質に対し過剰。 |
