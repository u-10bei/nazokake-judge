# U5 NFR Requirements — 出題停止（item retirement）

**ユニット**: U5（著作権配慮による出題停止）。FD 全 6 問 A（BR-U5-01〜13）/ NFR Req 全 5 問 A（2026-07-17）。
**性格**: 既存インフラ上の**列追加 2 本 + 読み取り経路の分割**。新規の公開面・外部依存・性能要求なし。NFR の重心は **「新規のみ反映／それまでの結果は有効」という要件の両輪を構造とテストで守れているか**に置く。

---

## Data / Migration

- **U5-NFR-01（migration 0004 の安全性）**: 列追加は `ALTER TABLE items ADD COLUMN retired_at TEXT` / `ALTER TABLE sessions ADD COLUMN likert_targets TEXT` の 2 本のみ。**いずれも NULL 許容**ゆえ **0002 のようなテーブル再構築は不要**・**既存行のデータ移送も不要**。適用直後の意味論は**既存 items = 全て現役（`retired_at IS NULL`）/ 既存 sessions = 全てフォールバック（`likert_targets IS NULL`）**＝**適用しただけでは何も変わらない**（安全な no-op 移行）。（Q1）
- **U5-NFR-02（適用順序の厳守）**: **migration → deploy** の順を厳守する（Infra §4 の既存規約）。逆順では `retired_at` 前提のコードが旧スキーマに当たる。`deploy.yml` は versioned 自動適用ゆえ **ワークフロー無変更**（`0001`〜`0004` が順に適用される）。（Q1）
- **U5-NFR-03（後方互換フォールバックの実装・検証必須）**: `likert_targets IS NULL` のセッションは **`list_items()`（全件）から Likert ターゲットを導出**し、**従来挙動を完全再現**する（BR-U5-04）。**本番は未デプロイ**ゆえ初回デプロイに 0004 を同梱すれば旧セッションは実在しないが、**フォールバックは実装し検証する**。理由: (i) 将来 U5 を**稼働後**に適用する場合は進行中セッションが必ず生じる、(ii) **フォールバックの有無が「新規のみ反映」の保証そのもの**、(iii) 実装コストが低く保険として合理的。（Q1）

## Compatibility（後方互換）

- **U5-NFR-04（エクスポート形式の凍結・U3/U4b 無変更）**: U5 は**エクスポートの形式を一切変えない**。`ExportItem` は `item_id`+`layer` のまま（**`retired_at` を出さない**）、**`EXPORT_FORMAT_VERSION` は 1.0.0 据え置き**（BR-U5-10）。
  - **保証手段 = 既存 PU3-3（自己完結性 PBT）+ 既存 U4b テスト群を無改修で緑に保つこと**（＝形式が変わっていない証拠）。
  - **⚠️ 検知器としての運用**: **U5 のために U3/U4b のテストを書き換えたら、それは形式を変えた証拠＝設計違反のシグナル**として扱う（テストの改変自体を異常検知に使う）。
  - 将来 retired の分析が必要になった時点で版上げ（1.1.0）+ U4b 改修を別途行う。（Q3）

## Testability（PBT 中心・強制セット PBT-02/03/07/08/09 への対応）

- **U5-NFR-05（PBT-03 = 不変条件・主対象）**: **要件の両輪それぞれに PBT を立てる**。FD で BR-U5-02 の禁止事項を明文化したが**明文は実装を強制しない**ため、テスト側にも網を張る:

  | ID | 不変条件 | 守る要件 | 根拠 |
  |---|---|---|---|
  | **PU5-1** | 廃止済み item X を含むプールで**新規セッション**を開始 → **ペア列・練習ペア・Likert ターゲットのいずれにも X が現れない** | 出題停止そのもの | BR-U5-02a/02b |
  | **PU5-2** | `likert_targets IS NULL` のセッションで、**廃止の前後で Likert ターゲットが一致** | **新規のみ反映**（片輪） | BR-U5-04 |
  | **PU5-3** | retire/unretire を**複数回適用しても状態が同一**・再廃止で**初回の `retired_at` が保持** | 冪等・証跡の非改竄 | BR-U5-06 |
  | **PU5-4** | 廃止の前後で **export の `items` 集合が不変**・**judgments の item ⊆ items（自己完結性）が保たれる** | **それまでの結果は有効**（片輪） | BR-U5-10 |

  - **PU5-4 は BR-U5-02 の直接の検出網**: `list_items()` に active フィルタを足す劣化実装をしたら**落ちる**。
  - **PU5-2 は「新規のみ反映」の直接の検出網**: 同じ劣化実装で**落ちる**。→ **両輪それぞれに独立した網**がかかる。（Q2）
- **U5-NFR-06（PBT-07 = ジェネレータ品質）**: PBT ジェネレータは **廃止済み / 現役が混在するプール**を生成する。「**廃止ゼロ件だけを引くジェネレータでは PU5-1/PU5-4 の反例探索が空回りする**」（U4b で単調性ジェネレータの適用範囲を誤り、ci profile の反例で初めて気づいた教訓の反映）。廃止件数・層構成・参照状況（`pairs` 参照の有無）を振る。（Q2）
- **U5-NFR-07（PBT-02 = ラウンドトリップ・U5 で新たに該当）**: `sessions.likert_targets` は **JSON 配列で保存・復元**する。**保存した配列と読み出した配列が順序を含めて一致**すること（Likert の提示順が意味を持つため順序保存が必須）。
  - U4b では PBT-02 を**非該当**（一方向変換）と明記したが、**U5 はシリアライズ/デシリアライズを新設するため該当**する。XC-02（セッション状態のラウンドトリップ）の延長線上にあり、既存方針（XC-02 = DB 行復元）と整合。
- **U5-NFR-08（unit = example の対象）**: PBT で扱わない個別挙動は unit で固定する: 参照済み item の廃止が**成功**する（凍結ガードを通らない, BR-U5-05）／`pool_ingest` 再投入で **`retired_at` 不変**（BR-U5-08）／`not_found`・`already_retired` の分類（BR-U5-06）／**充足を割ったら `token_issue` 拒否**（BR-U5-09）／`admin_log` の出力内容。（Q2）

## Observability / 監査（著作権対応ゆえ NFR に昇格）

- **U5-NFR-09（監査ログの必須化・廃止履歴の正）**: `admin_log` に **`item_retire` / `item_unretire`** を記録する。内容 = **対象 `item_id` 列挙・件数・結果**。**これが廃止履歴の正**（`retired_at` は現在状態のみを表し、unretire で前回の廃止時刻は失われる, BR-U5-13）。**本文（`body`）はログに出さない**（既存の秘匿方針を維持）。（Q4）
- **U5-NFR-10（証跡の残らない経路の排除）**: 廃止・復活は **API/CLI 経由を正**とする。**D1 直操作（`wrangler d1 execute` での UPDATE）は証跡が残らないため運用上排除**し、`runbook.md` に明記する。（Q4）

## Usability（CLI）

- **U5-NFR-11（`pool_retire` の出力と終了コード）**: 結果を **`retired` / `already_retired` / `not_found` に分類表示**する。
  - **終了コード**: 正常（部分的な `already_retired`・`not_found` を**含む**）は **0**。認証失敗・通信失敗・入力不正は **非 0**（U4a CLI の既存規約に合わせる）。
  - **`not_found` は失敗にしない**: 「既に存在しない＝目的は達成」であり、**冪等な再実行を失敗扱いにすると運用が回らない**。ただし **stderr に警告**を出す（タイポの気づきは確保）。（Q4）

## Security（既存流用・差分なし）

- **U5-NFR-12（Security 差分なし）**: retire/unretire は **既存 AuthGuard（Basic 認証）の背後**の `/admin/*`（U4a の単一チョークポイント）に置く。**新規シークレット・新規公開面・CORS 変更なし**。クエリは**全パラメータ化**（`item_id IN (...)` のバインド）。**参加者 API（`/api/*`）には一切触れない**。（Q5）

## 回帰

- **U5-NFR-13（回帰の完了基準・ブロッキング）**: **U1/U2/U3/U4a/U4b の既存 unit+PBT を全緑**に保つことを完了のブロッキング条件とする。
  - 特に **PU3-3（export 自己完結性）が緑 = BR-U5-02 の禁止事項を踏んでいない証拠**。
  - **integration**: U2/U3 の既存シナリオ（実 D1）を回し、**migration 0004 適用後も壊れない**ことを確認する。（Q5）

## 非目標（意図的にやらないこと）

| 項目 | 理由 |
|---|---|
| **即時停止（進行中セッションからの除去）** | 要件で「進行中セッションへの反映は不要」と確定（BR-U5-03）。進行中の露出は完了 or 非アクティブ 48h まで継続することを**受容済み**。必要になれば配信段のスキップを別ユニットで |
| **`retired_at` による廃止履歴** | カラムは**現在状態**のみ（BR-U5-13）。履歴は `admin_log` が担う |
| **`retired_at` のエクスポート出力** | 版上げ（1.1.0）+ U4b 波及を招く。将来必要になった時点で（BR-U5-10 / U5-NFR-04） |
| **物理削除** | `pairs` の FK 違反 + 自己完結性破壊（BR-U5-01） |
| **`retired_at` のインデックス** | プール約 95 件・全走査で十分（FD domain-entities §1） |
| **管理 UI からの廃止操作** | 研究者のプール管理は CLI が正（personas.md）。UI は監視・エクスポート専用 |
| **`Item` 契約への `retired_at` 追加** | 投入経路から廃止・復活を構造的に不可能にする（BR-U5-12） |

## Performance / Availability / Scalability / Resiliency
**すべて N/A**。プール約 95 件・UPDATE 数件・`WHERE retired_at IS NULL` の全走査。既存 Worker/D1 の運用に同じで U5 固有の要求なし。（Q5）
