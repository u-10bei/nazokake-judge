## 調査で判明した事実（実装確認済み・2026-07-20）

| # | 事実 | 影響 |
|---|---|---|
| 1 | **`items.layer` に CHECK 制約が実在**（`CHECK (layer IN ('pro','ai','edit','rule'))`, 0001 と 0002 の両方） | SQLite は CHECK を ALTER 不可 → **migration 0005 でテーブル再構築**（0002 と同型）。「データ投入だけ」では済まない |

| 4 | ⚠️ **`select_likert_targets` は層ラウンドロビン**（層バケット→ソート順に巡回） | 5 層化で Likert 10 枠が **2 枠/層**。下帯アンカーはプールの **5%（2/38）なのに Likert の 20%** を占め、**較正回帰が下帯に引っ張られる** → Q2 |

| 6 | **BR-01（同一ペア重複禁止）は `used: set[frozenset]` で構造的に保証済み** | 「同一評価者への同一ペア再提示ゼロ」は**既にハード制約**。仕様への明記のみで実装追加は不要 |
| 7 | **練習ペアは本番と同一の `generate_pairs` 呼び出し・同一プール由来** | 練習で見た item が本番にも出る＝**初見性が損なわれる** → Q5 |
| 8 | **`ExportJudgment` は `token` / `pair_index` を持つ** | **「出現回数」は分析時に導出可能**（token 内で pair_index 順に累積カウント）＝**スキーマ追加なしで共変量検証できる** |

### 事前生成の実現可能性（実測済み）

**J=228 は n=38 の 12-正則グラフ（辺数 38×12/2 = 228）と完全一致**する。実際に構成し、8 評価者へ配分できることを確認:

```
辺数 228 / 次数 min=12 max=12  → 露出の相対 gap = 0.000（完全均衡）
配分 [29,29,29,29,28,28,28,28] 成功
評価者内の同一項目 最大出現 = 3（k=3 以内）/ 同一ペア重複 = 0 件
```

**対比（オンライン方式・20 試行）**: 連結成分=1 は 20/20 で達成するが、相対 gap は平均 0.45・最悪 **0.66〜0.75**（露出 min=8 / max=16〜17 ＝**項目間で 2 倍の精度差**）。**較正 α=0.7 の境界上**であり、p=3・α=0.7・S=30 は n=95 での較正値ゆえ n=38 では保証されない。

→ **事前生成により較正は「無効化」ではなく「不要」になる**（オンライン重み付けを使わないため）。

## 質問（回答は各 `[Answer]:` に記入。特記なき場合 **★A=推奨デフォルト**）

### Q1【層値】下帯アンカーの enum 値をどうするか
`layer` の定義は**出自**（プロ作品層 / AI 生成層 / 編集・自作層 / ルールベース生成層）。一方「下帯アンカー」は**役割**（尺度の床を定義する指名アンカー）。S03・S13 は「整い○の人手古典」＝出自としては**人手・古典**であり、既存 4 層のいずれとも異なる。

- **★A（推奨）**: **`classic`**（値）/ 表示名「人手古典（下帯アンカー）」。**出自ベースの命名を維持**し、層の定義（＝出自）と整合させる。役割（下帯アンカー）は文書・分析側で扱う。層順序予測（機械組立 < 自由生成 < 編集 ≦ プロ）に `classic` を**含めない**ことを BR で明記（役割が違うため）。
- **B**: **`anchor`**（値）/「下帯アンカー」。役割ベース。直感的だが、**層＝出自の定義が崩れる**（将来「上帯アンカー」を作ると層が役割で増殖する）。
- **C**: `pro_low`。→ プロ層の一種に見え、**バー汚染の懸念が名前に残る**。不採用。

### Q2【★測定妥当性】下帯アンカーを Likert 較正アンカーに含めるか
**事実 #4**: `select_likert_targets` は層ラウンドロビン。5 層化すると Likert 10 枠を 2 枠/層で配分し、**下帯アンカー 2 件が両方とも較正アンカーになる可能性が高い**（プールの 5% が較正の 20%）。較正は (平均 Likert, θ) の単回帰ゆえ、**下帯 2 点が回帰直線を引っ張る**。

- **★A（推奨）**: **下帯アンカーを Likert 対象から除外**する（`select_likert_targets` は `classic` 層をスキップ）。理由: 下帯アンカーは **BT 尺度上の床を定義する指名アンカー**であり、その役割は θ の位置で果たされる。Likert 較正まで担わせると**同じ 2 件が二重に効く**（BT の床 + 較正の下端）。較正は残り 4 層（36 件）から選ぶ方が代表性が高い。
- **B**: プール比例配分に変更（10 枠を件数比で配分 → 下帯は 0〜1 枠）。→ 層ラウンドロビンの設計意図（層を均等にカバー）を変えることになり、他層の較正代表性も動く。
- **C**: 層均等のまま（下帯 2 枠）。→ **較正回帰が下帯に引っ張られる**。測定妥当性の観点で不採用。

### Q3【★事前生成】プランの生成・格納・引き当て方式
オンラインの `generate_pairs`（`session.py:53`）を、事前生成した割当プランの引き当てに置換する。

- **★A（推奨）**: **オフライン CLI で生成 → D1 に格納 → トークン発行時にプランを紐付け**
  - `scripts/plan_generate.py`: プール（active）と E・J から**均衡設計**を構成し、`assignment_plan` として D1 に投入。**決定論**（seed 記録）で監査再現可能。
  - **トークン発行時に `plan_index` を割り当てる**（`token_issue 8` → 各トークンに 0..7）。→ セッション開始時は**プランを引くだけ**（割当ロジックが実行時から消える）。
  - `start_or_resume` は `generate_pairs` の代わりに**プランを読み `save_pair_sequence` に流す**（原子保存は不変）。
  - **利点**: 設計が事前に固定＝**監査可能・再現可能**、露出 gap=0、較正が不要になる。
- **B**: セッション開始時にプランを決定論的に計算（事前生成せず、評価者インデックスから導出）。→ 事前に設計を検査できない（投入前に「本当に均衡か」を確認できない）。
- **C**: プランを JSON ファイルで保持し Worker が読む。→ D1 が正の設計に反する・配布経路が増える。

### Q4【運用リスク】E=8 でのドロップアウト
事前生成は「**全員が完走する**」前提で均衡が最適。E=8 では **1 名脱落＝12.5% 欠損**で、その評価者の 28〜29 ペアがまるごと欠け、均衡が崩れる（該当 item の m が 12→11 等に低下）。

- **★A（推奨）**: **予備プランを含めて生成**（例: E=10 分のプランを生成し、先着 8 名分を使用）。未使用プランは欠番として記録。**脱落時は予備トークンを配布**して補充する。均衡は「使用されたプランの集合」で再評価する（BT は欠損に強く、m が多少揺れても連結性は保たれる）。
- **B**: 脱落を許容し欠損のまま解析。→ m のばらつきが生じるが BT は成立する。運用は最も単純。
- **C**: 脱落判明時にプランを再生成。→ 既に回答済みのデータと整合させる必要があり複雑。

### Q5【初見性】練習ペアを事前生成の対象にするか / 本番項目との重複
**事実 #7**: 現状の練習ペアは本番と**同一プール・同一呼び出し**由来ゆえ、**練習で見た作品が本番にも出る**。ご指摘の「埋蔵性は初見でしか測れない（pivot は一度見たら掘り出し済み）」に照らすと、練習で消費された初見性は回復しない。

- **★A（推奨）**: **練習専用の作品を用意し、本番プール（n=38）から分離**する。練習用は `layer` を持つ必要があるが**分析対象外**（`is_practice=1` で出力段から除外済み）。→ **本番 38 件すべての初見性が保たれる**。練習用作品の追加投入が必要（3〜5 件程度）。
- **B**: 練習も事前生成に含め、**その評価者の本番ペアに出ない作品**を練習に充てる。→ 追加投入不要だが、n=38 のうち練習に使った分がその評価者の本番から欠け、**均衡設計が崩れる**（12-正則が壊れる）。
- **C**: 現状維持（練習も本番プールから・重複あり）。→ 初見性が損なわれる。ただし**出現回数を共変量として記録**すれば分析で統制可能（事実 #8）。

### Q6【既存資産】`generate_pairs` と α/S 較正をどうするか
- **★A（推奨）**: **`generate_pairs` は削除せず残す**が、**参加者フローからは呼ばれなくなる**（`session.py` の呼び出しのみ差し替え）。理由: (i) 既存 PBT（P-1 露出均衡・層間比率・k 制約）が**事前生成アルゴリズムの検証にも流用できる**、(ii) 将来 E を増やす/オンライン方式に戻す判断の余地を残す、(iii) 削除は回帰リスクを増やす。
  - **α/S 較正（p=3・α=0.7・S=30）は「参加者フローでは不使用」と明記**し、較正資産は `generate_pairs` に紐づく記録として保存する。
  - **事前生成プランには同等の PBT を課す**: 露出均衡（gap=0 を検証）・層間比率 ≥0.65・k≤3・**同一評価者内の同一ペアゼロ**・**全体の連結性**。
- **B**: `generate_pairs` を削除。→ 資産と検証を失う。将来の選択肢も消える。

### Q7【分析仕様】出現回数を共変量として記録・検証
**事実 #8**: `ExportJudgment` は `token` と `pair_index` を持つため、**「その評価者にとって当該 item が何回目の出現か」は分析時に導出可能**（token 内で pair_index 順に累積カウント）。**スキーマ追加は不要**。

- **★A（推奨）**: **スキーマは変更せず、導出方法と検証項目を分析仕様として文書化**する。BR に「出現回数（初見/再見）は `token`+`pair_index` から導出可能」と明記し、**「出現回数の主効果が β 推定に乗るか」を分析計画の必須確認項目**として `manual-p-rsch.md` に記載。
  - 併せて**同一ペア再提示ゼロがハード制約であること**（事実 #6）を BR に明記。
- **B**: `judgments` に出現回数カラムを追加。→ **導出可能な値を冗長に持つ**ことになり、整合性リスクを増やす。不採用。
