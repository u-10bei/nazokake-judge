### ★ 層間比率は「円周上の item 配置」で決まる（実測）

巡回グラフ `C38(1..6)`（228 辺）で、層の並べ方だけを変えた実測:

| 円周配置 | 層間ペア | 比率 | ゲート（≥0.65） |
|---|---:|---:|---|
| **層ごとに固める**（grouped） | 89 | **0.390** | ❌ **不足** |
| **層をインターリーブ** | 166 | **0.728** | ✅ 充足 |
| **シャッフル**（seed 固定） | 176 | **0.772** | ✅ 充足 |

→ **同じ辺集合・同じ正則次数でも、item の並べ方だけで層間比率がゲートを割る**。**配置は生成アルゴリズムの必須要件**であり、「たまたま通る」に委ねてはいけない（Q2）。

## 質問（回答は各 `[Answer]:` に記入。特記なき場合 **★A=推奨デフォルト**）

### Q1【`POOL_LAYERS` の配置と「非空要求」の分離】
**事実 #1** より、`plan_generate`（scripts）が層フィルタを使う以上、定数は **`schema` に置くしかない**（`backend` は scripts から見えない）。問うのは**責務の分割**。

- **★A（推奨）**: **`schema` に 2 つの定数を分けて置く**:
  - **`POOL_LAYERS`** = `(pro, ai, edit, rule, anchor)` — **充足判定の母数**（`practice` を除外）
  - **`REQUIRED_LAYERS`** = `(pro, ai, edit, rule)` — **非空を要求する層**
  - **分ける理由**: 両者は**別の概念**。`anchor` は**母数には入る**（層間ペアの相手になる・出題される）が、**非空を強制すると `anchor` 不在のプール（ドライラン・将来の構成変更）でゲートが落ちる**。下帯アンカーは**研究上の要請**であって**割当アルゴリズムの成立条件ではない**（4 層あれば層間ペアは組める）。
  - `pool_sufficiency` は **`POOL_LAYERS` で母数を絞り、`REQUIRED_LAYERS` で非空を検査**する。
- **B**: `POOL_LAYERS` 1 つで両方を兼ねる（`anchor` の非空も要求）。→ 単純だが、**ドライランで `anchor` を用意しないとゲートが落ちる**。`anchor` の欠落を「アルゴリズムの不成立」と扱うのは意味論的に不正確。
- **C**: `backend/domain` に置く。→ **`scripts` から import できない**（事実 #1）。不採用。

### Q2【★プラン生成】構成による保証と生成後検証の切り分け
BR-U6-10 の①〜⑥を、**構成で保証する**ものと**生成後に検証する**ものに分ける。

- **★A（推奨）**: **構成で保証できるものは構成で保証し、残りを検証する**:

  | 制約 | 保証方法 |
  |---|---|
  | ① 露出 gap=0 | **構成**（m-正則グラフ＝全頂点が次数 m） |
  | ② 全体連結 | **構成**（巡回グラフ `C_n(1..d)` は距離 1 の輪を含むため必ず連結・事実 #3） |
  | ④ 同一ペア 0 | **構成**（全辺が相異なる＝集合として構成） |
  | ⑤ 層間比率 ≥0.65 | **構成（配置）+ 検証**。**層をインターリーブして円周に配置**する（実測: grouped 0.390 ❌ / interleave 0.728 ✅）。配置は必須要件とし、**結果は必ず検証**する |
  | ③ k 制約 ≤3 | **検証**（分割の貪欲配分に依存）。満たすまで**seed を進めて再試行**（決定論は「seed → プラン」で保つ） |
  | ⑥ ブロック連結 | **検証**（分割に依存・BR-U6-20）。同上 |

  - **生成 CLI は「構成 → 検証 → 失敗なら seed を進めて再試行 → 上限回数で明示失敗」**とする。**決定論は保たれる**（同一 seed から同じ試行列を辿る）。
  - **①②④を構成で押さえるのが要**: 検証だけに頼ると「反例が出たが直し方が分からない」状態になる。
- **B**: すべて探索（ランダム生成 + 検証 + リトライ）。→ **層間比率が配置依存であることを見落とすと永久に収束しない**（grouped 配置では 0.390 で構造的に不可能）。
- **C**: すべて構成で保証（分割まで解析的に）。→ k 制約とブロック連結を同時に満たす分割の解析的構成は過剰。

### Q3【引き当ての LC 構造】プランをどこで読み、フォールバックをどこで分岐するか
`session.py:53`（`generate_pairs` の唯一の呼び出し）を置換する。

- **★A（推奨）**: **Repository に読み取りを、サービス層に分岐を置く**:
  - `Repository.get_active_plan_set()` → 有効な `plan_set`（`is_active=1`）
  - `Repository.get_plan_pairs(plan_set, plan_index)` → そのスロットのペア列（練習含む）
  - `Repository.get_token_plan_index(token)` → `plan_index`（NULL 可）
  - **`session.start_or_resume` の新規分岐で**: `plan_index` が **NULL なら従来の `generate_pairs`**（フォールバック, U6-NFR-14）/ **非 NULL ならプランを引く**。
  - **`save_pair_sequence` 以降は一切変更しない**（Session + PairSequence + `likert_targets` の同一 batch 原子保存＝U5 DP-U5-02 を維持）。
  - **`generate_pairs` は無改修で残る**（BR-U6-17）。呼ばれるのは**フォールバック経路のみ**。
- **B**: プラン引き当てを `domain` に置く。→ **domain が repo に依存**＝層の逆流。不採用。
- **C**: `session.py` が D1 を直接触る。→ Repository を I/O 境界に保つ既存方針に反する。

### Q4【脱落時の引き継ぎ粒度】（FD からの持ち越し・BR-U6-15）
補充トークンが**何を引き継ぐか**を確定する。

- **★A（推奨）**: **未消化ペアのみを引き継ぐ**。補充トークンには**元トークンが未回答のペアだけ**を `pairs` として作る。
  - **理由**: **これが m=12 を保つ唯一の方法**。スロット全体をやり直すと**既回答分のペアが二重に判定され、該当 item の比較数が 12 を超えて露出 gap≠0 になる**（＝事前生成の前提そのものを壊す）。
  - 未回答ペアは `judgments`（token, pair_id）との差分で導出できる。
  - **評価者効果の分析時に「当該スロットは 2 名」と注記**（BR-U6-15）。
- **B**: スロット全体をやり直す。→ **露出均衡が壊れる**（上記）。不採用。
- **C**: 脱落分は欠損のまま（補充しない）。→ 該当 item の m が 12→11 等に低下。BT は成立するが均衡が崩れる。**補充できるならすべき**。

### Q5【activate ガードの実装形態 + 適用性評価】
- **★A（推奨）**:
  - **`activate` 切替の拒否は「アプリ層の事前検証」**で実装する（U6-NFR-20）。**判定条件**: **有効セットに紐づく judgment が 1 件でも存在したら拒否**。
    - **DB 制約では表現できない**（`judgments` と `assignment_plan_meta` をまたぐ条件）。
    - **拒否は 4xx で明示**し `admin_log` に記録（何を試みて拒否されたかが証跡に残る）。
  - **`is_active` の一意性**は **`assignment_plan_meta` の部分ユニークインデックス** or アプリ検証（NFR Design Part 2 で実装形態を確定）。
  - **適用性**: キャッシュ / キュー / CB / 分散ロック / スケール = **N/A**（プラン引き当ては PK 参照・運用者の逐次操作）。**インデックスは `assignment_plan` の PK のみ**（プール 38・プラン 228 行×2 セット）。
- **B**: activate 拒否を実装しない（運用注意のみ）。→ **実験の同一性に直結する操作**を無防備にする。不採用。
