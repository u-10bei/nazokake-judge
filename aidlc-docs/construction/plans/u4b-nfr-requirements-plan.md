# U4b NFR Requirements Plan — BT 集計スクリプト（bt_aggregate）

**ユニット**: U4b（C-SCRIPT-BT・最終ユニット）
**目的**: U4b の非機能要件を確定する。U4b は**オフライン pure-Python スクリプト**（Worker/D1 非依存・ローカル/CI 実行）ゆえ、通常のセキュリティ/可用性/データ移行 NFR は N/A 中心。固有論点は **統計推定の再現性・数値安定性**と **PBT 中心のテスト戦略**。推定の正しさ（MM 擬似データ・識別可能性・較正）は Functional Design（BR-U4b-01〜13）で決定済みなので NFR として明文化する。
**前提（既決）**: 拡張 opt-in は U1〜U3 と共通（Security Baseline=No / Resiliency=No / **PBT=Partial**、強制 PBT-02/03/07/08/09）。追加依存なし（標準ライブラリのみ, BR-U4b-13）。入力は U3 ExportBundle（版検証 BR-U4b-11）。

このドキュメントは **Part 1（Plan + 質問）**。回答・承認後に `nfr-requirements.md`（U4b-NFR-NN）/ `tech-stack-decisions.md`（U4b 追加分）を生成します。

## 質問（回答は各 `[Answer]:` に記入。特記なき場合 **★A=推奨デフォルト**）

### Q1【再現性・数値安定性】BT 推定の決定論と数値健全性
- **★A（推奨）**: (i) **完全な決定論**（固定初期値 π=1・固定反復順・パラメータ〈α, 収束閾値, 最大反復〉既定固定 → 同一入力同一 BTResult。乱数不使用, P-6/BR-U4b-01）。(ii) **収束の明示**（`converged`/`iterations` を BTResult に。未収束は warnings）。(iii) **数値安定性**: π は正値維持・log/除算のゼロ割回避（正則化 α で分母>0 保証, BR-U4b-03）・θ 正規化は最大連結成分内（BR-U4b-04）。(iv) パラメータ（α・閾値・最大反復）は CLI で上書き可・既定値を記録（監査）。
- **B**: パラメータをハードコード・収束情報を出さない。→ 再現性・監査性が下がる。

[Answer]: A ＋ 決定論を一段強化: **「同一入力ファイル→同一出力」ではなく「同一データ（行順序不問）→同一出力」**。推定前に items・観測ペア・judgments 集計を **item_id で正準ソート**してから反復する。純 Python の浮動小数演算は演算順序依存ゆえ、同じ判定データでも D1 エクスポート行順が変わると Σ の加算順が変わり θ が最下位桁で揺れうる。反復判定装置＝複数スナップショット突合の運用で、行順序という非本質的差でスコアが揺れない方が診断上安全。実装は集計後のソート数行、**PU4b-2 の PBT に置換不変性（judgments をシャッフルしても BTResult 一致）を追加**して反例探索で担保。(ii)(iii)(iv) は A のとおり。**未収束時の挙動を明文化**: `converged=false` + warnings で結果は出力し**終了コード 0**（エラーにしない。最大反復到達でも θ は使用可能な近似・収束フラグで研究者が判断）。

### Q2【Testability】PBT/単体の振り分け
- **★A（推奨）**: **BT 推定の不変条件は PBT（Hypothesis）**（U4b は純関数中心＝PBT が主, PBT-03 系）:
  - PU4b-1 単調性（A 常勝→θ_A>θ_B、**擬似データ正則化 ON でも保存**）、PU4b-2 決定論、PU4b-3 正規化（**最大連結成分内 Σθ=0**）、PU4b-4 識別可能性（非連結→警告+最大成分・除外 item 可視化）、PU4b-5 較正係数復元（既知線形データ・example+PBT）。
  - **単体（example）**: CLI 入出力・版検証（PU4b-7）・U3 突合（PU4b-6・同一エクスポートで matches/wins 一致）・出力形式。
  - **PBT ジェネレータ**: プール・比較列（連結/非連結を含む）・Likert アンカーを生成。強制セット中 U4b の主対象は **PBT-03**（不変条件）。ラウンドトリップ（PBT-02）は U4b 非該当（明記）。
- **B**: example のみ。→ 推定不変条件の反例探索が弱い（BT は境界条件が多い）。

[Answer]: A ＋ **PU4b-2 に置換不変性プロパティを追加**（judgments シャッフルで BTResult 一致＝Q1 の行順序不問決定論）。PBT-02（ラウンドトリップ）非該当の明記も適切（U4b は入力→出力の一方向変換で逆写像を持たない）。ジェネレータは **「非連結グラフを意図的に含む」生成が PU4b-4 の反例探索の要**ゆえ、連結/非連結を両方生成する設計を NFR に明記。

### Q3【Performance】MM 収束時間の SLO 姿勢
- **★A（推奨）**: **明示的な数値 SLO は設定しない**（U1〜U3 と同方針）。参考目安「実行 < 数秒」。MM は本規模（items 約 95・判定数千・観測ペア限定）で純 Python でも数秒未満収束。反復上限で無限ループを防ぐ（既定十分大）。numpy 化・並列は**不要**（BR-U4b-13）。
- **B**: 数値 SLO・numpy 最適化を要件化。→ 規模に対し過剰。

[Answer]: A — 数値 SLO なし・参考目安「数秒未満」・反復上限で無限ループ防止・numpy 化不要。U1〜U3 と同方針で一貫。

### Q4【Usability】CLI・出力の可用性
- **★A（推奨）**: CLI は **引数=エクスポート JSON パス**、オプション `--out`（出力ファイル）・`--alpha`・`--max-iter`・`--tol`・`--allow-version-mismatch`。**JSON（機械可読）+ 人間可読テーブル（層別・スコア降順）** を出力。**warnings を目立つ形で表示**（非連結・較正スキップ・版不一致緩和・除外 item）。日本語で要点。終了コードは正常 0 / 版不一致エラー 非0。
- **B**: JSON のみ・警告は JSON 内のみ。→ 研究者が異常（非連結等）を見落とす。

[Answer]: A ＋ **終了コード規定を補完**: 版不一致（非0）に加え、**入力ファイル不在・JSON パース不能・ExportBundle 検証失敗も非0**（CI 自動実行で失敗を exit code 検知できることが必要）。**warnings 系（非連結・較正スキップ・未収束）は 0 のまま**。オプション体系・日本語 warnings・JSON+テーブル併記は A のとおり。

### Q5【Security】オフライン・トークン付きデータの取扱
- **★A（推奨）**: U4b は**オフライン**（ネットワーク不要・入力は研究者自身のエクスポート）。エクスポートは**トークン付き研究データ**（U3-NFR-03）だが、U4b は**それをローカルで読むだけ**で新たな公開面を作らない。BTResult は**集計値**（item 単位スコア）でトークンを含めない（token は BT 入力に使うが出力には出さない＝再識別の緩和）。エクスポート/結果ファイルのローカル保管は運用責任（NFR-08、リポジトリ管理外）。SQLi/認証等は非該当（D1・API なし）。
- **B**: BTResult に token 別内訳も出す。→ 集計目的に不要な再識別リスク。集計値のみで足りる。

[Answer]: A ＋ 事実是正: **BT 推定も Likert 較正も token を一切使わない**（推定は item_left/item_right/choice、較正は target_ref/rating のみ）。正確には「**U4b は token を読み飛ばし、出力にも含めない**」。NFR 文書では「**BTResult は token 非含有かつ U4b の計算経路自体が token 非参照**」と記す方が防御として強い（A の再識別緩和をさらに強める事実）。ローカル保管の運用責任・新規公開面なしは A のとおり。
