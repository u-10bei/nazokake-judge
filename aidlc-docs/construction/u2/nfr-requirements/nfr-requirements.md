# U2 NFR Requirements — 参加者セッション（participant）

U1 の U1-NFR-01〜15・U4a の U4a-NFR-01〜12 を前提に、U2 固有の非機能要件を **U2-NFR-NN** で定義する。U2 は**参加者に直接触れる初の UI + 公開 API** であり、最大論点は (1) トークン=資格の衛生、(2) 出自秘匿による評価健全性、(3) モバイル/日本語 UI の可用性。
**技術決定の詳細**: `tech-stack-decisions.md`（TSD-U2-01〜06）を参照。

---

## 1. セキュリティ衛生（最重要 / Security 拡張は No・通常実装で必須）

- **U2-NFR-01（トークン=資格・接頭辞分離）**: 参加者 API は `/api/*`（`/admin/*` と分離）。**認証はトークン自体**（Basic 認証なし）。無効/存在しないトークンは各エンドポイントで拒否。HTTPS 強制。（Q3, BR-U2-26）
- **U2-NFR-02（キャッシュ滞留防止）**: **全 `/api/*` 応答に `Cache-Control: no-store`**。トークン・謎かけ本文がプロキシ/ブラウザキャッシュに残らないようにする。（Q3, BR-U2-27）
- **U2-NFR-03（ログ秘匿）**: **参加者系ログにトークン生値・謎かけ本文を出力しない**（AdminLog=U4a-NFR-03 と同水準）。ログ相関にトークンの非可逆表現を使う場合は **ハッシュ形式を単一規約に固定**（例: SHA-256 先頭 8 文字）。これにより特定参加者フローを生値なしで追跡でき、デバッグ可用性と秘匿を両立。（Q3 補足）
- **U2-NFR-04（トークン受け渡し）**: POST 系（judgment/likert/survey）の**トークンは body 渡し**で統一。クエリ渡しは起動時（配布 URL 由来 `GET /api/session?token=`）のみ。URL/クエリ経路の露出を最小化。（Q3, BR-U2-27）
- **U2-NFR-05（基本衛生）**: 全 D1 アクセスはパラメータ化クエリ（SQLi 対策, BR-12/U1-NFR-07）。API 入力は schema/ Pydantic で検証（choice ∈ {A,B}・rating ∈ [1,7]・pair 帰属・フェーズ整合はサーバが最終判定, BR-U2-11/12/18）。（XC-03, Requirements Q12=B）

## 2. 研究健全性（Security 派生・U2 固有）

- **U2-NFR-06（出自秘匿＝ブラインド評価の強制）**: 参加者向け API レスポンスは**本文（`body`）と表示に必要な最小項目のみ**を返し、**`layer`（プロ/AI 等）・`body_ref`・`seed`・`exposure_snapshot` を一切含めない**。`ItemView = {item_id, body}` を **API 契約レベルで固定**する。他参加者分・発行総数も参加者スコープ外。（Q4, domain-entities §4）
  - **分類の意図**: FD の設計判断を「**研究データ健全性を守るセキュリティ要件**」に昇格＝削ってはいけない要件化。「デバッグ用に layer を返す」等のフラグ出し分け（事故の入口）を**明示的に不採用**とし、ブラインド評価の成立条件を契約で守る。

## 3. 可用性（Usability・U2 固有・XC-04）

- **U2-NFR-07（モバイルファースト・日本語）**: 主要画面（判定・Likert・アンケート）は**モバイルファースト・レスポンシブ**（A/B 縦積みで両作品が読みやすい・十分な行間/フォント・大きめタップ領域）、**日本語のみ**（切替なし）。進捗は本番判定のみをモバイル視認位置に表示（BR-U2-13）。（Q5, XC-04）
- **U2-NFR-08（アクセシビリティ水準）**: **合理的水準**（セマンティック HTML・十分なコントラスト・キーボード操作可）を満たす。**正式な WCAG 2.1 準拠適合は目標としない**（統制された評価者集団・一般公開サービスではないため）。（Q5）
- **U2-NFR-09（サーバ権威・楽観更新なし）**: 送信は**サーバ応答（SessionView/SubmitResult）を待って次画面へ**遷移し、**楽観的 UI 更新はしない**。送信中/失敗/再試行は非ブロッキング表示。フェーズ遷移の真実は常にサーバ（BR-U2-03/29 と整合）。（Q5）

## 4. 信頼性・データ整合（Reliability / Integrity）

- **U2-NFR-10（冪等は DB 側で保証）**: 判定は `(token,pair_id)` 一意（既存 DP-02）、**Likert は `(token,target_ref)` 一意（migration 0003）** で **初回不変**（`ON CONFLICT DO NOTHING` + 既存値返却）、**Survey は PK=token で upsert**。アプリ層の check-then-insert に依存しない（競合窓を残さない, U1-NFR-04 の一貫適用）。（Q7, BR-U2-11/17/21）
- **U2-NFR-11（完了順序保証）**: `mark_token_completed` は **本番判定全件 ∧ Likert 全対象評定 ∧ survey 行あり** をサーバ確認後にのみ実行（一方向 BR-09）。いずれか未完なら completed にしない。（Q なし=FD 決定, BR-U2-24）
- **U2-NFR-12（同時開始の露出競合を許容）**: 複数参加者の同時新規開始で露出スナップショットが競合しても**ロック・直列化を設けない**。鮮度ズレは XC-01 累積収束（P-1）で吸収、`exposure_snapshot` は「実際に参照した値」として保存され監査リプレイの完全性は保たれる。（Q2, U1-NFR 非目標と一貫）
- **U2-NFR-13（再送セマンティクス・トークン無期限）**: 正しさは**サーバ冪等が権威**。クライアント再送は UX 目的の軽いリトライ（指数バックオフ程度）で、二重登録は原理的に起きない。at-least-once/exactly-once の追加インフラは設けない。**トークンに有効期限を設けない**（in_progress は非アクティブでも再開可能）。※**非アクティブ 48h の露出集計除外（BR-04）はトークン失効ではない**（集計対象の話であり、離脱した参加者は後日再開できる）。（Q8）

## 5. 性能（Performance・最小限）

- **U2-NFR-14（SLO 非設定）**: 参加者向けレイテンシに**明示的な数値 SLO を設定しない**（U1-NFR-01/02 と同方針）。参考目安として非公式に「**セッション開始 < 1s・各送信 < 500ms**（体感即時）」を置く。想定規模（〜50 セッション × 約 43 ペア ≈ 2,000 行）では露出の毎回導出（`read_exposure_counts`）も D1 上で実質瞬時であり、キャッシュ・マテリアライズは不要。（Q1）

## 6. 可観測性（Observability・最小限）

- U1 LogEmitter（`emit`）を再利用し、参加者フローの主要イベント（セッション開始・完了遷移・無効/完了トークン拒否・業務エラー）を構造化出力。**トークン生値・本文は非出力**（U2-NFR-03）。監視基盤なし（stdout, U1-NFR-10 と同方針）。相関はハッシュ規約（U2-NFR-03）で。

## 7. テスト容易性（Testability）

- **PBT（`tests/pbt/`, 純粋ロジック）**: PU2-1（`serialize`/`deserialize` ラウンドトリップ, PBT-02）／PU2-6（`select_likert_targets` 決定論・fixed 包含・層網羅・件数, PBT-03 系）／PU2-3（`derive_phase` 単調性=純粋述語ゆえ PBT 可能）。PBT 強制セットは U1/U4a と同一（PBT-02/03/07/08/09）。（Q6）
- **integration（`tests/integration/`, 実 D1・`/api/*` 越し）**: PU2-2（再開の非重複）／PU2-4（判定冪等）／PU2-5（練習の集計除外整合）／PU2-7（Likert 初回不変）／PU2-8（完了順序保証）。統合ハーネスは U1/U4a のものを流用し参加者フロー一巡を検証。（Q6）

## 8. データ・マイグレーション

- **U2-NFR-15（migration 0003 + params 拡張）**: `migrations/0003_likert_unique.sql` を **versioned** で追加（`likert_responses` に `UNIQUE(token, target_ref)`）。**新規プロジェクトで既存行なし**のため安全。`AssignmentParams.likert_fixed_targets: list[str] | None = None` を `schema/` に追加。適用は `wrangler d1 migrations`（dev→prod、**適用→デプロイの順**）、0002 と同じ流儀。schema/ とテストを同時更新（U2 スコープ）。（Q7, BR-U2-15/17）

---

## 9. 非目標（意図的な非要件）

- **Scalability / Resiliency**: N/A（単独研究者・小規模、総参加者数十名・同時数名）。高可用性・自動フェイルオーバー・DR は設計しない（NFR-06）。
- **数値 SLO**: 設定しない（U2-NFR-14）。計測・追跡の運用コストが小規模の価値を上回る。
- **セッション開始の直列化（ロック）**: 設けない（U2-NFR-12）。露出競合は累積収束で吸収。
- **at-least-once/exactly-once の追加インフラ**: 設けない（U2-NFR-13）。DB 一意制約で冪等を担保。
- **トークン有効期限**: 設けない（U2-NFR-13）。BR-04（露出集計除外）は失効ではない。
- **正式 WCAG 準拠適合**: 目標としない（U2-NFR-08）。合理的アクセシビリティ水準に留める。
- これらを「要件を課さない」意図的非要件として明示記録（後の再燃防止。U1/U4a と同じ流儀）。
