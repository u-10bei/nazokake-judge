# U6 Business Logic Model — 層拡張 + 事前生成割当

**中核**: 割当ロジックを**実行時から設計時へ移す**。オンラインの重み付きランダム抽選を、**事前生成・検査済みの固定プランの引き当て**に置換する。

---

## 1. 設計の移動（Before / After）

```
【Before: オンライン重み付きランダム】
  セッション開始 ──▶ read_exposure_counts（全 pairs から露出導出）
                 └─▶ generate_pairs(pool, exposure, seed, params)   ← 実行時に設計
                     ├ 重み 1/(露出+1)³ で抽選（α=0.7 / S=30 の較正が前提）
                     └ 制約: k≤3 / 層間 0.65 / 同一ペア禁止
                 └─▶ save_pair_sequence
  ⚠️ 事前に設計を検査できない。露出 gap は 0.45〜0.75（項目間で 2 倍の精度差）
  ⚠️ 較正は n=95 での値。n=38 では保証されない

【After: 事前生成プラン】
  ①設計時（オフライン CLI・投入前に検査可能）
     plan_generate ──▶ 12-正則グラフ（38 頂点・228 辺）を決定論構成
                   ──▶ 8 スロットへ分割 [29,29,29,29,28,28,28,28]
                   ──▶ BR-U6-10 の①〜⑤を検証 ──▶ assignment_plan として D1 投入
                       （plan_set = 成立版 / フォールバック版 の 2 セット）
  ②発行時
     token_issue 8 ──▶ 各トークンに plan_index 0..7 を割当
  ③実行時（割当ロジックなし）
     セッション開始 ──▶ プランを引く ──▶ save_pair_sequence（原子保存は不変）
  ✅ 露出 gap = 0（全 item ちょうど m=12）／較正は不要／設計が監査可能
```

**置換点は `session.py:53` の 1 箇所のみ**（`generate_pairs` の呼び出しはここだけ）。`save_pair_sequence` 以降の原子保存・`likert_targets` の同一 batch 保存（U5 DP-U5-02）は**一切変更しない**。

## 2. なぜ完全静的なのか（FR-03 との関係）

```
FR-03（要件 L52・非目標表 L162）
  「適応的サンプリングは v1 では採用せず、将来の反復判定フェーズで再検討する」
  検証質問の選択肢 C =「これまでの結果をもとに情報量の多いペアを動的に選ぶ」→ 不採用
  確定 = Q2=X（計画的制約付きランダム）
        │
        ▼
「ブロック1の結果でブロック2を再生成」= 選択肢 C そのもの
        │
        ▼
完全静的化は **新たな設計変更ではなく既決要件への復帰**（BR-U6-13）
```

**廃止対象の切り分け**:

| | 内容 | 静的設計との関係 |
|---|---|---|
| **(a) 設計の再生成** | ブロック 1 の結果でブロック 2 の**ペアを組み直す** | **廃止**（FR-03 が既に不採用） |
| **(b) 推定の逐次更新** | ブロック 1 で暫定 BT → ブロック 2 込みで再推定（**分析側のみ**） | **両立・廃止不要** |
| **トークン発行 4+4** | 段階的な募集ペース | **維持**（設計に影響しない） |

## 3. 読み取り経路（U5 の二本立てを継承）

U5 で確立した「**これから出題するものを選ぶ**なら active /**既に配ったもの**を解決するなら全件」の原理は不変。U6 はここに **`POOL_LAYERS` による層フィルタ**を重ねる。

| 用途 | 参照先 |
|---|---|
| **プラン生成**（設計時・CLI） | `list_active_items()` の **`POOL_LAYERS` 該当分のみ**（＝`practice` を除外） |
| **充足判定** | 同上（**`practice` は母数に入れない**, BR-U6-05） |
| **`build_view` の bodies** | `list_items()`（**全件**・練習素材も含む＝練習ペアの本文解決に必須） |
| **エクスポート `items`** | 全件（自己完結性 BR-U3-07・U5 から不変） |

> ⚠️ **`practice` 層を `POOL_LAYERS` から外すことが本設計の要**。外さないと「練習素材が本番プールの充足母数に入る」「`anchor`/`practice` の非空を機械的に要求する」誤動作が生じる（BR-U6-05）。

## 4. プラン生成フロー（設計時）

```
入力: プール（active ∩ POOL_LAYERS）= 38 件 / E=8 / J=228 / seed
  │
  ├─▶ ① 12-正則グラフを構成（38 頂点・各次数 12・辺数 228）
  │      → 露出は構成上ちょうど m=12（gap=0 が設計で保証される）
  │
  ├─▶ ② 8 スロットへ分割 [29,29,29,29,28,28,28,28]
  │      制約: 評価者内の同一 item 出現 ≤ 3（k 制約）
  │
  ├─▶ ③ 練習ペアを全スロット共通で固定記載（開示セット・is_practice=1）
  │      → 本番 38 件は一切使わない（初見性の保全, BR-U6-16）
  │
  ├─▶ ④ 検証（BR-U6-10）: 露出 gap=0 / 連結成分=1 / k≤3 / 同一ペア0 / 層間≥0.65
  │      ✗ なら投入せず失敗（**投入前に検出できることが事前生成の価値**）
  │
  └─▶ ⑤ assignment_plan として D1 投入（plan_set・seed を記録）
```

**2 プランセット**（BR-U6-12）: 成立版（N1〜N8 あり）とフォールバック版（N1〜N8 を置換）を**両方生成・両方投入**し、**許諾判明後に一方を activate**。activate は**実験開始前に確定**し、収集開始後は切り替えない。

## 5. 実行時フロー（割当ロジックなし）

```
参加者が URL を開く
  │
  ▼
start_or_resume(token)
  ├─ 既存セッションあり → 従来どおり再開（U2 から不変）
  └─ 新規:
       ├─▶ トークンの plan_index を取得
       ├─▶ activate 中の plan_set × plan_index のペア列を引く（練習 + 本番）
       ├─▶ Likert ターゲット = プラン付属の固定リスト（BR-U6-06）
       └─▶ save_pair_sequence（Session + PairSequence + likert_targets を同一 batch, U5 DP-U5-02）
```

**実行時に抽選は起きない。** `read_exposure_counts` も新規セッションでは不要になる（露出は設計時に確定済み）。

## 6. スロット運用（ドロップアウト）

```
plan_index=3 の評価者が 15/29 で離脱
  │
  ├─ ❌ 予備プラン方式は採らない（BR-U6-14）
  │     E=10 分を生成して 8 枠のみ消化 → 44 辺が未消化 → 12-正則が壊れる
  │     ＝均衡の前提そのものを破壊する
  │
  └─ ✅ 同一 plan_index の補充トークンを発行（BR-U6-15）
        └─▶ 未消化の 14 ペアを引き継ぐ
            judgments は token 単位ゆえ **BT 集計は不変**（item の m=12 が保たれる）
            評価者効果の分析時に「当該スロットは 2 名」と注記
```

**引き継ぎの粒度**（未消化分のみ再開 / スロット全体やり直し）は **NFR Design 以降の決定点**。

## 7. 分析仕様（スキーマ追加なし）

| 項目 | 導出方法 |
|---|---|
| **出現回数**（初見/再見） | `token` 内で `pair_index` 順に item の累積出現をカウント（**`ExportJudgment` の既存フィールドのみで可能**, BR-U6-19） |
| **同一ペア再提示** | **ゼロがハード制約**（設計段階で保証・全 228 辺が相異なる, BR-U6-18） |
| **バー（プロ水準）** | **指名アンカー（S04・S10・N 群）の β 位置**。層平均は使わない（BR-U6-03） |
| **層順序予測** | `rule < ai < edit ≦ pro` の 4 層のみで検証。**`anchor` は含めない**（BR-U6-02） |

**分析計画の必須確認項目**: 「**出現回数の主効果が β 推定に乗るか**」（BR-U6-19）。初見/再見の系統差は交絡であると同時に、**再見は読み返し基準側に寄る**という codebook との整合もあるため、共変量として明示的に扱う。

## 8. 受入基準（Given-When-Then）

- Given `layer='anchor'` の item、When `pool_ingest`、Then **投入成功**（0005 の CHECK に追加済み）。
- Given `practice` 層のみ空、When `token_issue`、Then **充足ゲートに影響せず発行成功**（`POOL_LAYERS` 外, BR-U6-05）。
- Given 検証済みプラン、When 新規セッション開始 ×8、Then **全 item の出現回数がちょうど 12**・**連結成分 1**・**同一ペア重複 0**。
- Given 練習ペア、When 参加者が本番に進む、Then **練習で見た作品が本番に一切現れない**（BR-U6-16）。
- Given Likert 固定リスト 10 件、When セッション開始、Then **ラウンドロビンが走らず指名 10 件がそのまま対象**（BR-U6-06）。
- Given plan_index=3 の脱落、When 補充トークンを発行、Then **未消化ペアを引き継ぎ item の m=12 が保たれる**（BR-U6-15）。
- Given 成立版/フォールバック版の両投入、When 一方を activate、Then **activate 版のみが引き当てられる**（BR-U6-12）。
