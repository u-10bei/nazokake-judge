# U6 NFR Requirements — 層拡張 + 事前生成割当

**ユニット**: U6。FD 全 7 問確定（BR-U6-01〜20）/ NFR Req 全 5 問 A + 追加 7 点（2026-07-20）。
**性格**: 本ユニットは **`items` テーブル再構築 + 参加者フローの割当置換**という**これまでで最も侵襲的な変更**。NFR の重心は **migration の安全性**と**事前生成プランの検証**に置く。

---

## Data / Migration（★最重要・実測でリスク顕在化）

- **U6-NFR-01（0005 は子行退避方式・単純再構築は失敗する）**: `items` の再構築は **`pairs` の子行を退避してから**行う。
  - **実測事実**: `pairs` が `items` を FK 参照する行を持つ状態では **`DROP TABLE items` が `FOREIGN KEY constraint failed`** になる。0002 が通ったのは「新規プロジェクトで既存行なし」だったためで、**0005 は初めて「データがある状態での親テーブル再構築」**になる。
  - **`PRAGMA foreign_keys=OFF` / `defer_foreign_keys=ON` は D1 の migration 実行環境では効かない**（両方実測で失敗）。
  - **手順**: `CREATE TABLE pairs_bak AS SELECT * FROM pairs` → `DELETE FROM pairs` → `items` 再構築 → **列を明示して** `pairs` へ復元 → `DROP TABLE pairs_bak`。
- **U6-NFR-02（`retired_at` の引き継ぎ必須）**: 再構築時に **U5（0004）の `retired_at` 列を必ず引き継ぐ**。列を落とすと**廃止状態が消失**し、著作権対応の現在状態が失われる。**Code Gen の Step に一行固定**する。
- **U6-NFR-03（列を明示して復元）**: `pairs` の復元は **`SELECT *` に依存せず全列を明示**する（列順・列追加の変化に強くする）。
- **U6-NFR-04（★適用ウィンドウの前提条件・ブロッキング）**: 退避方式は **`DELETE FROM pairs` → 復元の間に「空の `pairs` を読む窓」**が生じる。稼働中に適用すると**露出計算・セッション再開処理が壊れる**。
  → **0005 の適用は「発行済み未消化トークンが存在しない時点」に限る**ことを**ブロッキング前提**として明記する。**実験開始前のカットオーバーで自然に満たせる制約**（コストゼロ）。
- **U6-NFR-05（適用後検証の定型化）**: 0005 適用後に次を**必ず**確認する（実測でやった検証をそのまま手順化。`dry-run-dev.md` に定型として記載）:
  1. `PRAGMA foreign_key_check` が**違反なし**
  2. **`items` / `pairs` の行数が適用前後で一致**
  3. **`retired_at` 非 NULL 件数が適用前後で一致**
- **U6-NFR-06（FK 全数調査結果を DDL に残す）**: 0005 のヘッダコメントに**調査結果**を記載する（**0004 が採った様式の踏襲**）:
  - `items` を参照する FK は **`pairs.item_left` / `pairs.item_right` の 2 本のみ**
  - **`likert_responses.target_ref` は FK 非設定**（素の TEXT）
  - **`judgments` は `tokens` のみ参照**（`pair_id` は FK ではない）
  → **退避対象が `pairs` だけで足りる根拠**を DDL 自身が持つ。
- **U6-NFR-07（`practice` 層の同載）**: `practice` は **0005 に同載**する（後から足すと**テーブル再構築をもう一度**やることになる）。
- **U6-NFR-08（原子性の活用）**: **D1 は migration を原子的にロールバックする**（実測: 失敗時に `items_new` の残骸なし）。→ 適用失敗時は**中途半端な状態にならない**ことを前提にしてよい。

## Testability（PBT 中心）

- **U6-NFR-09（プラン生成器の PBT・PBT-03 が主）**:

  | ID | 不変条件 | 対応 BR |
  |---|---|---|
  | **PU6-1** | 全 item の出現がちょうど m（**露出 gap = 0**） | BR-U6-10 ① |
  | **PU6-2** | プラン**全体**の比較グラフの連結成分 = 1 | BR-U6-10 ② |
  | **PU6-3** | 評価者内の同一 item 出現 ≤ k | BR-U6-10 ③ |
  | **PU6-4** | 同一評価者に同一ペアが現れない | BR-U6-10 ④ / BR-U6-18 |
  | **PU6-5** | 層間ペア比率 ≥ 0.65 | BR-U6-10 ⑤ |
  | **PU6-6** | 同一 (プール, E, J, seed) → **同一プラン**（決定論） | BR-U6-11 |
  | **PU6-7** | **★各ブロック（4 スロット）の辺和集合だけで連結成分 = 1** | **BR-U6-10 ⑥ / BR-U6-20** |

  **PU6-7 の必要性**: **BR-U6-13 (b) 推定の逐次更新の前提**。ブロック1 単体で暫定 BT を出すには**ブロック1 の比較グラフが連結**でなければならない（非連結なら最大成分しか推定されず全項目の暫定位置が出ない）。**実測では貪欲配分 30 試行でブロック1・2 とも成分 1 だが構成上の保証ではない**（116 辺 ≫ 連結に必要な 37 辺）ため、E・分割・構成法の変更で静かに壊れうる。→ **明示的な検証項目とする**。

- **U6-NFR-10（ジェネレータ品質・PBT-07）**: PBT ジェネレータは **n・E・J を振って生成**する。**n=38/E=8/J=228 の 1 点だけでは「その組合せでたまたま通る」ことしか示せない**（U4b で単調性ジェネレータの適用範囲を誤った教訓）。
- **U6-NFR-11（失敗系の明示）**: 次の入力で**生成が明示的に失敗**すること:
  1. **正則グラフが構成できない組合せ**（`2J` が `n` で割り切れない等）
  2. **J の分割指定の総和 ≠ J**（[29×4, 28×4] 等を引数化する場合の入力検証）
- **U6-NFR-12（unit の対象）**: `POOL_LAYERS` による層フィルタ（**`practice` が充足母数外**）/ `plan_index` の引き当て / 補充トークンの引き継ぎ / `admin_log` の出力内容。
- **U6-NFR-13（integration の対象・実 D1）**: **0005 を「データがある状態」で適用**（U6-NFR-01/04/05）・プラン投入 → セッション開始 → **ペア列がプランと一致** ・**U2/U3/U4a/U5 の既存シナリオが緑**。

## Compatibility（後方互換）

- **U6-NFR-14（`plan_index IS NULL` のフォールバック）**: `plan_index` が NULL のトークンは**従来どおりオンライン生成にフォールバック**する（U6 以前のトークン／ドライラン用の即席トークン）。**実装し検証する**（U5 の `likert_targets IS NULL` フォールバックと同型）。
- **U6-NFR-15（エクスポート形式の凍結・U3/U4b 無変更）**: `EXPORT_FORMAT_VERSION` は **1.0.0 据え置き**。`ExportItem.layer: str` は値域を列挙していないため**新層値がそのまま流れる**。
  - **保証手段 = U3/U4b の既存テストを無改修で緑に保つこと**（形式が変わっていない証拠、**U5-NFR-04 と同型**）。
  - **⚠️ 検知器としての運用**: **U6 のために U3/U4b のテストを書き換えたら設計違反のシグナル**。
- **U6-NFR-16（U5 資産の保全）**: 0005 再構築後も **`retired_at` が保持**され、**`list_active_items()` / `pool_retire` が従来どおり動く**ことを検証する。**U5 の BR-U5-02（`list_items()` 凍結）も維持**。

## Observability / 監査

- **U6-NFR-17（プラン投入・activate は管理 API 経由）**: `POST /admin/plan`（投入）/ `POST /admin/plan/activate`（有効化）を**既存 AuthGuard の背後**に置く。**D1 直操作は証跡が残らないため運用上排除**（`pool_ingest`・`pool_retire` と同じ流儀、U5-NFR-10 と同型）。
- **U6-NFR-18（★証跡はプランの「内容」に紐づける）**: `admin_log` の `plan_ingest` / `plan_activate` には **`plan_set` 名だけでなく `seed` + プラン内容ハッシュ**を記録する。
  → 「どのセットを有効化したか」が**名前ではなく内容に紐づき**、事前記録（タスク5 §7）との照合が**改竄不能**になる。プラン切替は**実験の同一性に直結する**操作ゆえ、名前だけの証跡では不十分。
- **U6-NFR-19（activate は 1 セットのみ）**: 有効なプランセットは常に 1 つ（BR-U6-12）。
- **U6-NFR-20（収集開始後の activate 切替は拒否・推奨を先行記録）**: **有効セットに judgment が 1 件でも存在したら activate 切替を拒否（ハード）**。切替が必要な事態は**実験の作り直し**であり、**API で簡便にやれてはいけない操作**。→ 実装形態（DB 制約 / アプリ検証）は **NFR Design の決定点**だが、**方針はここで固定**する。

## Security（既存流用）

- **U6-NFR-21**: 新規公開面は **`/admin/*` 配下のみ**（既存 Basic 認証背後）。**参加者 API（`/api/*`）に新規公開面なし**。クエリは**全パラメータ化**。新規シークレットなし・CORS 変更なし。

## 回帰

- **U6-NFR-22（回帰の完了基準・ブロッキング）**: **U1/U2/U3/U4a/U4b/U5 の既存 unit+PBT を全緑**に保つ。**integration は 0005 適用後に U2/U3/U4a/U5 の既存シナリオを実 D1 で緑**にする。**`generate_pairs` の既存 PBT（P-1）も緑のまま維持**（BR-U6-17 で残す方針ゆえ）。

## 非目標（意図的にやらないこと）

| 項目 | 理由 |
|---|---|
| **★フォールバック経路の統計的性質の保証** | `plan_index IS NULL` のオンライン生成は **dev/ドライラン専用**。**n=38 プール + 5 層値環境での露出均衡・層間比率は保証外**（較正値 p=3/α=0.7/S=30 は **n=95 由来**）。**本実験データに使わないこと**は U6-NFR-17/18 の activate 証跡側で担保する（＝どのプランで収集したかが記録に残る） |
| **適応的サンプリング（設計の再生成）** | **FR-03 が既に不採用**（BR-U6-13）。**(b) 推定の逐次更新は存続**（分析側） |
| **`retired_at` / `plan_index` のエクスポート出力** | 版上げ（1.1.0）+ U3/U4b 波及を招く。**token の別で評価者は識別可能**ゆえ `plan_index` は不要 |
| **予備プラン（E>8 分の生成）** | J=228 は 8 スロットへの完全分割ゆえ**均衡が壊れる**（BR-U6-14） |
| **`generate_pairs` の削除** | 資産・既存 PBT・将来の選択肢を残す（BR-U6-17） |
| **`retired_at` / `layer` のインデックス** | プール 38 件・全走査で十分 |
| **Performance / Availability / Scalability / Resiliency の SLO** | プラン引き当ては PK 参照。**実行時の抽選がなくなるため軽くなる** |
