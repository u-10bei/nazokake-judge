# U6 NFR Design Patterns — 層拡張 + 事前生成割当

U6-NFR-01〜22 を設計パターン **DP-U6-NN** に落とす。U6 の設計は **「割当を実行時から設計時へ移す」構造**に集約される。新奇な部品は導入せず、既存の層構造（`scripts` → `schema` / `backend` → `domain`）に収める。NFR Design 全 5 問 A + 追加 5 点（2026-07-20）。

---

## DP-U6-01: 設計時への移動（実行時ロジックの除去）

割当を**実行時の抽選**から**設計時の固定プラン**へ移す。

```
Before: セッション開始 ─▶ 露出導出 ─▶ 重み付き抽選（較正 α/S が前提）─▶ 保存
        ⚠️ 事前に設計を検査できない / 露出 gap 0.45〜0.75 / 較正は n=95 由来

After:  設計時 ─▶ 構成 + 制約付き探索 + 検証 ─▶ D1 へ投入（検査済み）
        実行時 ─▶ プランを引く ─▶ 保存（抽選なし）
        ✅ 露出 gap=0 / 較正が不要 / 投入前に設計を検査できる
```

- **置換点は `session.py:53` の 1 箇所**（`generate_pairs` の唯一の呼び出し）。**`save_pair_sequence` 以降は一切変更しない**（U5 DP-U5-02 の原子保存を維持）。
- **`generate_pairs` は無改修で残す**（BR-U6-17）。呼ばれるのは**フォールバック経路のみ**。
- 対応: U6-NFR-14, BR-U6-09/17, Q3。

## DP-U6-02: 構成で保証するもの / 検証するもの / 探索で作るものの三分割（★中核）

制約を**どう満たすか**で三分割する。「全部検証で担保」でも「全部構成で保証」でもない。

| 制約 | 方法 | 根拠 |
|---|---|---|
| ① 露出 gap=0 | **構成** | m-正則グラフ＝全頂点が次数 m |
| ② 全体連結 | **構成** | 巡回グラフ `C_n(1..d)` は距離 1 の輪を含むため**必ず連結** |
| ④ 同一ペア 0 | **構成** | 全辺が相異なる（集合として構成） |
| ⑤ 層間比率 ≥0.65 | **探索（配置）+ 検証** | **配置依存**（実測: grouped **0.390 ❌** / interleave 0.728 / shuffle 0.772） |
| 内容制約①〜③（禁止/忌避/濃縮） | **探索（配置）+ 検証** | 円周距離で表現（DP-U6-03） |
| ③ k 制約 ≤3 | **検証 + seed 再試行** | 分割依存 |
| ⑥ ブロック連結 | **検証 + seed 再試行** | 分割依存（BR-U6-20） |
| 内容制約④（提示順） | **順序付け + 検証** | 辺集合でなく**ペア列の順序**（DP-U6-03） |

- **①②④を構成で押さえるのが要**: 検証だけに頼ると「**反例は出たが直し方が分からない**」状態になる。
- **⑤を探索に置くのが要**: **grouped 配置では 0.390 で構造的に不可能**ゆえ、「ランダム生成 + リトライ」では**永久に収束しない**。**配置は必須要件**であって「たまたま通る」に委ねられない。
- **決定論は「seed → プラン」で保つ**（再試行は同一 seed から同じ試行列を辿る）。
- 対応: U6-NFR-09/10/11, BR-U6-10, Q2。

## DP-U6-03: 内容制約を「円周配置 + 順序付け」に写す（★研究要件の受け皿）

形式制約（グラフ理論）だけでは**タスク5 §3 の内容制約を表現できない**。**巡回グラフの幾何**に写して扱う（BR-U6-21）。

| 種別 | 写し方 |
|---|---|
| **辺の禁止（ハード）** | **円周距離 >6 に配置** → `C_n(1..6)` の隣接は距離 ≤6 ゆえ**構成的に排除**。**PU6-8 で不在を検証** |
| **辺の忌避（ソフト）** | 同上を**目的関数の一項**に。**排除できなくても失敗させず**レポートに残す |
| **辺の濃縮** | 対象 item を**円周上で近接配置** → 12 辺のうち何本を対象に充てるかは配置で決まる。**目的関数の一項** |
| **提示順の回避** | **分割後に順序付け（sequencing）ステップ**を 1 段追加。同一評価者内で指定グループを連続させない |

- **器と中身の分離**: **制約の中身は研究側（タスク5）の成果物、器は U6**。→ **CLI が制約ファイルを入力に取る**（禁止ペア・忌避ペア・濃縮目標・隣接回避グループ）。研究側の更新が**コード変更なしに反映**される。
- **実測（成立性）**: 禁止辺 8 本・濃縮対象 9 件で近傍探索（seed 固定）→ **禁止辺違反 0 / 層間 0.706 / 濃縮 9-9 本を同時達成**。**3 目的は両立する**。
- **提示順は `pair_index` として運ばれる**: プランのペア列がそのまま `save_pair_sequence` に渡るため、順序制約は実行時に再現される（Q3 と整合）。
- 対応: BR-U6-21, U6-NFR-09（PU6-8）, Q2。

## DP-U6-04: 明示リストによる暗黙走査の置換 + 概念の分離

`for layer in Layer` の**enum 全走査**をやめ、**明示定数**に対して検査する。

- **`POOL_LAYERS`** = `(pro, ai, edit, rule, anchor)` — **充足判定の母数**（`practice` を除外）
- **`REQUIRED_LAYERS`** = `(pro, ai, edit, rule)` — **非空を要求する層**

**2 つに分ける理由**: 両者は**別の概念**。`anchor` は**母数には入る**（層間ペアの相手になり出題される）が、**非空を強制すると `anchor` 不在のプールでゲートが落ちる**。**下帯アンカーは研究上の要請であって割当アルゴリズムの成立条件ではない**（4 層あれば層間ペアは組める）。

- **配置は `schema`**（`scripts` は `backend` を import できないため・実装確認済み）。
- **`pool_sufficiency` は「充足判定の唯一の実装」**ゆえ置換は**この 1 関数内で完結**させる（述語の単一性を壊さない）。
- **裏返しの手当て（DP-U6-05）**: 非空要求から外した分、**`anchor` 投入忘れは CLI 側で検出**する。
- 対応: U6-NFR-12, BR-U6-05, Q1。

## DP-U6-05: 研究側の入力検証を CLI に集約する

**ランタイムのゲートは「アルゴリズムの成立条件」だけを見る**。**研究上の組成要件は生成 CLI で検証**する。

- **期待組成**（`n=38` / 層別件数 10/2/14/9/3）を**引数で受け取り、実プールとの不一致で明示失敗**（BR-U6-22）。
- **理由**: `anchor` を `REQUIRED_LAYERS` から外した裏返しとして、**`anchor` 投入忘れのまま本番プランが生成される事故をランタイムが拾わなくなる**。**正しい置き場はランタイムではなく `plan_generate`**（制約ファイルと同じ「研究側入力」の層）。
- **結果**: ランタイムは**軽く保たれ**、研究要件の変更は**CLI 引数・制約ファイルの変更で閉じる**。
- 対応: BR-U6-22, Q1。

## DP-U6-06: 危険操作のガードと競合窓の除去

**プラン切替は実験の同一性に直結する**ため、二重に守る。

1. **収集開始後の activate 切替を拒否**（U6-NFR-20）: **有効セットに judgment が 1 件でも存在したら拒否（ハード）**。
   - **アプリ層の事前検証**で実装（`judgments` と `assignment_plan_meta` をまたぐ条件ゆえ **DB 制約では表現できない**）。
   - **4xx で明示 + `admin_log` に記録**（何を試みて拒否されたかが証跡に残る）。
2. **★トークンへの `plan_set` 束縛で競合窓を消す**: **発行時に `(plan_set, plan_index)` の組をトークンへ束縛**し、**セッション開始はトークン自身の `plan_set` を読む**（「その時点の有効セット」を参照しない）。
   - **除去される窓**: トークンが `plan_index` のみ持つ設計だと、**発行 → セッション開始の間に activate が切り替わるとトークンの意味が変わる**。
   - **副次効果**: activate ガードの述語が「**当該セットを参照するトークン / judgment の存在**」として**一意に定まる**。さらに「**どちらのセットで発行したか**」が**トークン単位で証跡に残る**。
- 対応: U6-NFR-17〜20, Q5。

## DP-U6-07: 証跡は「内容」に紐づける

`admin_log` の `plan_ingest` / `plan_activate` には **`plan_set` 名だけでなく `seed` + プラン内容ハッシュ**を記録する（U6-NFR-18）。

- **名前だけの証跡では改竄・取り違えを検出できない**。**内容ハッシュ**なら事前記録（タスク5 §7）との照合が**改竄不能**になる。
- **監査再現用に「初期 seed + 成功試行番号（または最終有効 seed）」の両方**をプランメタに記録する（再試行を挟むため、初期 seed だけでは再現に試行回数の情報が要る）。
- 対応: U6-NFR-18, BR-U6-11, Q2 追加。

## DP-U6-08: migration の子行退避（データがある状態での親再構築）

`items` の CHECK 制約更新は**テーブル再構築**を要するが、**`pairs` が FK 参照する行があると `DROP TABLE` が失敗する**（実測）。

- **`PRAGMA foreign_keys=OFF` / `defer_foreign_keys=ON` は D1 の migration 実行環境では効かない**（両方実測で失敗）。
- **子行退避方式**: `pairs_bak` へ退避 → `DELETE FROM pairs` → 親を再構築 → **列を明示して**復元 → `pairs_bak` 破棄。
- **`retired_at`（U5/0004）の引き継ぎ必須**（列を落とすと廃止状態が消失）。
- **適用ウィンドウの制約**: 退避中は**空の `pairs` を読む窓**が生じるため、**発行済み未消化トークンが存在しない時点に限る**（U6-NFR-04）。
- 対応: U6-NFR-01〜08。

## 導入しない設計部品（意図的な非採用・U1〜U5 と同方針）

| 部品 | 非採用理由 |
|---|---|
| キャッシュ | プラン引き当ては PK 参照・プール 38 件。 |
| キュー / 非同期 | 運用者の逐次操作。 |
| サーキットブレーカ / リトライ（実行時） | 外部依存なし。**再試行は設計時（生成 CLI）にのみ存在**。 |
| 分散ロック | 管理 API の逐次操作。activate の競合は**トークンへの `plan_set` 束縛**で構造的に消す（DP-U6-06）。 |
| **`retired_at` / `layer` / `plan_index` のインデックス** | プール 38 件・プラン 228 行×2 セット。PK で十分。 |
| 新規ライブラリ / 新規サービス | 標準 `random` / `hashlib` / `json` のみ。 |
| **実行時の割当ロジック** | **本ユニットで除去するもの**（DP-U6-01）。フォールバック経路にのみ残る。 |
