# データ提供元向け説明書（プール素材の編纂・提供）

**役割**: 判定装置に載せる**刺激プール（作品集合）と、その扱いに関する制約**を用意して運用者へ渡す。
**本書の位置づけ**: **何をどの形式で渡すか**と、**渡した後に何をしてはいけないか**を定める。
装置の運用手順は `runbook.md`、結果の読み方は `manual-p-rsch.md`（本書の読者の担当ではない）。

---

## 0. 渡すもの — 4 点

| # | ファイル | 内容 | 現状 |
|---|---|---|---|
| ① | `items_*.json` | **作品プール本体**（本文を含む） | ⛔ 要作成 |
| ② | `practice.json` | **練習ペア**（開示セット） | ⛔ 要作成・**忘れると練習なしで本番に入る** |
| ③ | `constraints.json` | 内容制約と Likert 固定 10 件 | ⛔ 雛形のまま（`plans/<set>/`） |
| ④ | `composition.json` | 期待組成（n・層別内訳） | ✅ **記入済み**（確認だけ） |

④ は設計側で確定済みです。**①〜③ が提供元の成果物**です。

---

## 1. `items_*.json` — 作品プール

JSON 配列 または JSONL。1 レコード = 1 作品。

```json
[
  {"item_id": "N01", "layer": "pro",    "body": "〇〇とかけまして△△と解く。その心は……", "body_ref": "ねづっち・出典メモ"},
  {"item_id": "S03", "layer": "anchor", "body": "…"},
  {"item_id": "E01", "layer": "edit",   "body": "…"},
  {"item_id": "PR1", "layer": "practice","body": "…"}
]
```

| フィールド | 必須 | 規約 |
|---|---|---|
| `item_id` | ✅ | **1 文字以上・プール内で一意**。**後から変更しない**（§3） |
| `layer` | ✅ | 6 値のいずれか（§2） |
| `body` | ✅ | 作品本文。**1 文字以上**。改行可 |
| `body_ref` | 任意 | 出自メモ（出典・制作日・制作条件など）。**参加者にも解析にも出ません**——提供元と運用者のための備考欄です |

**不正なレコードはそれだけが拒否**され、理由つきで返ります（`層ラベル/本文必須`）。全体が止まる
わけではないので、投入結果の `rejected` を必ず確認してください。

---

## 2. 層ラベル（6 値）

**`pro` / `ai` / `edit` / `rule` は「出自」**、**`anchor` は「役割」**、**`practice` は「用途」**です。
性質が違うものが同じフィールドに同居しているので、割当を間違えると解析が歪みます。

| 値 | 意味 | 解析での扱い |
|---|---|---|
| `pro` | プロ作品 | 層順序予測に**含める** |
| `ai` | AI 生成 | 〃 |
| `edit` | 編集・自作 | 〃 |
| `rule` | ルールベース生成 | 〃 |
| **`anchor`** | **下帯アンカー**（尺度の床を定義する指名作品） | ❌ **層順序予測に含めない**（出自ではなく役割ゆえ） |
| **`practice`** | **練習専用** | ❌ 分析的に不活性。**出力段で除外**され、プール充足の**母数にも入らない** |

### 迷ったときの判断

- **「プロが作ったが、尺度の床として指名した作品」→ `anchor`。** `pro` に入れるとプロ層の平均が
  下がり、層順序予測が壊れます。
- **「本番でも使いたい練習素材」は作れません。** `practice` は母数外・出力段除外なので、
  本番プールと**別に用意**してください。練習で見た作品が本番に出ると**初見性が失われます**。

**⚠️ 参加者に層構成を知らせてはいけません**（要求特性でデータが汚染されます）。装置側は
構造で守っています——本文は解析データに出ず、参加者 API は層を返しません。**残る穴は人間側の
説明**です。募集文で「AI と人間の比較です」と言った時点で崩れます。

---

## 3. `item_id` の規約 — ★最重要

**`item_id` は作品の恒久的な識別子**です。プラン（誰が何と何を比べるか）も、判定データも、
すべて `item_id` で作品を指します。**本文ではなく ID が作品の同一性を担っています。**

- プール内で**一意**
- **一度渡したら変更しない**（連番の振り直し・接頭辞の変更を含む）
- 意味のある命名を推奨（`N01`〜`N08` = ねづっち作、`S03` = 下帯アンカー、など）

`item_id` を変えると、**プラン投入時に「参照 item がプールに不在」で弾かれます**（安全側に
倒れる）。危ないのは次章です。

---

## 4. ★ 投入後の凍結 — `item_id` を保ったまま本文を直さない

**推敲・誤字修正のつもりの操作が、最も静かにデータを壊します。**

装置には凍結ガードがあり、**参加者が 1 人でも開始した後は、その作品への更新を拒否**します
（`参照済み item への更新は拒否（プール凍結）`）。ただし**ガードが効き始めるのは参加開始後**です。

### 差し替えが通ってしまう窓（実測）

| タイミング | 本文の差し替え | 
|---|---|
| プール投入 〜 **プラン確定前** | ✅ 通る（**ここは正常な修正機会**） |
| **プラン確定後 〜 参加開始前** | ⚠️ **通ってしまう**（`updated=1`） |
| 参加開始後 | ⛔ 拒否される |

問題は真ん中です。**プランは `item_id` しか持たない**ため、本文を差し替えても——

- プランの内容ハッシュは**変わりません**（照合を通ります）
- コミット済みの `plan.json` も**変わりません**

つまり **「コミットされたものが投入された」という証跡は成立しているのに、判定される作品が
すり替わっている**状態が作れてしまいます。装置はこれを検出できません。

**規約: プランを生成した後は、`item_id` を保ったまま `body` と `layer` を変更しない。**
修正が必要になったら、**運用者に申し出てプランを作り直してください**（プランはプールから
構成されるので、プール確定 → プラン生成の順序をやり直すのが正しい手当てです）。

> この窓は U6（事前生成割当）で構造的に長くなりました。生成 → コミット → 投入 → 有効化 →
> トークン発行 → 配布、という手順が挟まるためです。運用規約で閉じています。

**著作権上の理由で作品を止めたい場合**は差し替えではなく**出題停止**（`pool_retire`）を使います。
物理削除ではなく、**それまでの判定結果は有効のまま**新規セッションに出なくなります。
→ 運用者へ依頼してください（`runbook.md` §2-⑥-b）。

## 5. `practice.json` — 練習ペア（★忘れやすい）

**`item_id` のペアの配列**です。ここに挙げたペアが**全評価者共通で、本番の前に提示**されます。

```json
[["PR1", "PR2"], ["PR3", "PR4"], ["PR5", "PR6"]]
```

- 標準は **3 ペア**（= `practice` 層の作品が 6 件必要）
- **⚠️ 省略するとエラーにならず「練習なし」のプランが生成されます。** 参加者は練習を経ずに
  本番に入り、読み返しテストの習得がないまま判定します——**測定妥当性が落ちます**
- 練習ペアは**開示セット**です。参加者への説明・練習画面の趣旨と整合させてください

---

## 6. `constraints.json` — 内容制約と Likert 固定 10 件

**記入仕様は `plans/README.md`** にまとめてあります（本書では重複させません）。要点だけ:

- `likert_targets` は**ちょうど 10 件**。**この 10 件が全評価者に共通で提示**され、
  BT スコアを Likert 尺度へ写す**較正点**になります。**`practice` 層の作品は入れないでください**
- `forbidden_pairs` は**ハード制約**（違反すれば生成が失敗する）
- `discouraged_pairs` / `enrichment` は**ソフト**（違反してもレポートされるだけ）
- **未知キーは拒否**されます——`forbidden_pair` と単数で書くと typo で制約が黙って無効化される
  のを防ぐためです
- **セット別で流用できません**（`plan_set` が期待組成と一致しないと明示失敗）

**内容制約が無くても生成は通ります。** まず `likert_targets` の 10 件だけ埋めて他を `[]` にした
状態で通し、それから制約を足すのが安全です（記入ミスと生成の難しさを切り分けられます）。

## 7. ファイルの取り扱い

**作品本文はリポジトリに入りません。** `.gitignore` は**ファイル名で**除外しているので、
**名前の規約を必ず守ってください**。

| 用途 | ✅ 安全な名前 | ⚠️ 追跡されてしまう名前 |
|---|---|---|
| プール | `items_*.json` / `pool_*.json` | `data.json`・`works.json` |

```bash
git check-ignore -v items_real.json && echo "OK: 除外される"
```

何も出力されなければ**追跡対象**です。名前を変えてください。

一方 **`constraints.json` と生成されたプランはコミットします**——`item_id` しか含まず本文が
無いためです。これは要件で、**コミット履歴と内容ハッシュが「どのプランで実験したか」の証跡**に
なります。

## 8. 受け渡し前のセルフチェック

- [ ] `item_id` がプール内で**一意**
- [ ] `layer` が 6 値のいずれか。**`anchor` を `pro` に混ぜていない**
- [ ] 層別の件数が `composition.json` と**一致**（不一致なら生成時に明示失敗します）
- [ ] `practice` 層の作品が**本番プールと重複していない**
- [ ] **`practice.json` を用意した**（標準 3 ペア）
- [ ] `likert_targets` が**ちょうど 10 件**・すべてプールに実在・`practice` を含まない
- [ ] `forbidden_pairs` などの `item_id` が**すべて実在**（typo は自己ペアなどで検出されます）
- [ ] ファイル名が `items_*.json`（`git check-ignore` で確認済み）
- [ ] **プラン生成後に本文を直さない**ことを了解している（§4）
