# 研究者向け説明書（P-RSCH「オペレーター」）

**役割**: 実験を設計・運用し、回答データを回収して Bradley–Terry 推定を行う。
**本書の位置づけ**: **装置の全体像・判断の拠り所・結果の読み方**をまとめる。**具体的なコマンド手順は `runbook.md`**（本書はそこへの導線を張る／手順を重複させない）。参加者へ渡す文書は `manual-p-eval.md`。

---

## 1. この装置は何か

「**制作物がプロ水準に対しどこに位置するか**」を反復的に判定する装置です。人が 2 作品を比較した判定を集め、Bradley–Terry モデルで**1 本の尺度**に載せ、新作の相対位置を読みます。

```
① プール投入 → ② トークン発行 → ③ 配布 → ④ 参加 →
⑤ 進捗監視 → ⑥ エクスポート → ⑦ BT 集計 → 新作の位置確認
                                        ↺ 反復（スナップショットごと）
```

**なぜ比較判定なのか**: 「この作品は 5 点満点で何点か」を人に直接聞くと、評価者ごとに基準がずれます。「A と B のどちらが優れているか」なら基準のずれに強く、多数の比較を集めれば BT が**共通尺度上の強度 θ** を復元します。

## 3. 実験設計上の 3 つの約束

### ① 出自秘匿 — 参加者に層を知らせない

刺激プールは 4 層（`pro` / `ai` / `edit` / `rule`）ですが、**参加者はこの構成を知ってはいけません**。知ると判定が出自の推測に引きずられ（要求特性）、データが汚染されます。

- システム側は構造で守っています: **作品本文（`body`）はエクスポート経路に出ない**・参加者 API は層を返さない・管理 UI は公開アセットに置かない。
- **人間側の運用が最後の穴**です。募集文・案内・口頭説明で「AI と人間の比較です」と言った時点で崩れます。配布物は `manual-p-eval.md` をそのまま使ってください（層に一切触れない設計です）。

### ② 練習は集計に入らない

各セッション冒頭の練習ペアは**出力段（U3 のエクスポート）で除外済み**です。BT 集計は再フィルタしません。あなたが練習を除く操作をする必要はありません。

### ③ 割当は露出均衡・層間比率を保証する

ペア割当は、特定作品の比較回数が偏らないよう露出を均衡させ、層間ペア（異なる出自どうしの比較）を一定割合含めます（既定 65%）。これが**比較グラフの連結性**を担保し、BT 推定の精度を均一に保ちます。→ 偏りが心配なら `/admin/winrates` の `matches` を見てください。

## 4. 発行前に満たすべき充足条件

トークン発行時にプール充足がゲートされます（未充足なら発行拒否・exit 1）。既定パラメータ（`session_pairs=40`, `k=3`, `cross=0.65`）での具体値:

| # | 条件 | 既定での具体値 |
|---|---|---|
| ① | 総数 ≥ `ceil(2×session_pairs/k)` | **総数 ≥ 27** |
| ② | 4 層すべて非空 | pro/ai/edit/rule 各 1 件以上 |
| ③ | `(総数−最大層件数)×k ≥ ceil(cross×session_pairs)` | `(総数−最大層)×3 ≥ 26` |

③ は「**最大層に偏りすぎると層間ペアを作れない**」という意味です。1 層だけ大量に入れても発行できません。想定運用（4 層計 90〜95 件）なら余裕を持って満たします。

**段階投入は可能**です。未充足でも投入自体は成功し警告が出ます（充足は発行時にのみ強制）。

## 5. 結果の読み方（ここが本書の核心）

### 暫定勝率（`/admin/winrates`）と BT（`bt_result.json`）は別物

| | 暫定勝率 | BT スコア |
|---|---|---|
| 定義 | 勝ち数 ÷ 出場数 | 比較構造全体から推定した強度 θ |
| 用途 | **収集途中の傾向をざっと見る** | **正式な結論** |
| 弱点 | 「誰と当たったか」を無視する（弱い相手にばかり勝った作品が過大評価される） | — |

**論文に載せるのは BT です。** 暫定勝率は進捗確認用と割り切ってください。

### BTResult の各フィールド

| フィールド | 読み方 |
|---|---|
| `bt_score`（θ） | BT 尺度上の位置。**最大連結成分内で Σθ=0** に正規化＝**0 が推定対象の平均**。正なら平均より上 |
| `calibrated_score` | θ を Likert 相当尺度へ写像した値。**解釈しやすい単位**（例: 5.0 なら「Likert 5 相当」）。較正できなければ `null` |
| `rank` | 推定対象内の順位（θ 降順・同値は item_id 昇順） |
| `component` | 所属する比較グラフの連結成分。**0 = 最大成分 = 推定対象** |
| `matches` / `wins` | 生の観測カウント（**暫定勝率と同じ定義**なので `/admin/winrates` と突合できます） |
| `converged` / `iterations` | 収束したか。`false` でも結果は出ます |
| `source.exported_at` | **どのスナップショット由来か**。反復運用での取り違え防止 |

### 🚫 やってはいけない読み方

- **`component` が異なる item 同士のスコアを比較しない。** 比較グラフが分断されている＝両者を橋渡しする判定が存在しない＝**スコアは原理的に比較不能**です。並べて大小を論じると誤りになります。
- **`bt_score=null` を「弱い」と読まない。** 推定対象外（別成分 or 未出場）という意味です。データ不足であって、評価が低いのではありません。

### 「非連結」warning が出たら

比較グラフが分断されています。判定数を増やすと繋がります。**新作が別成分に落ちていたら、その新作の位置は読めません**（装置の目的が果たせていない状態）。層間ペア比率が効いているので通常は起きませんが、投入直後や回答数が少ない段階では起こりえます。

### α 感度チェック（論文向け）

正則化 α は「観測の少ないペアほど強く平均へ縮める」効果を持ちます。既定 α=1.0 は**観測 1 回のペアで擬似データが実データと同量**＝比較的強い縮小です。

**実データでは α ∈ {0.5, 1.0, 2.0} で回して順位の頑健性を確認してください。** 結論が α に依存しないことを示せれば、そのまま論文付録の頑健性記述になります（`BTResult.alpha` に使用値が記録されます）。

→ **コマンド**: `runbook.md` §2-⑦

## 6. あなたの Pain Point への対処

| 懸念 | どう解決されているか |
|---|---|
| **エクスポート形式と集計スクリプトの入力形式がずれて手作業変換が発生する** | `ExportBundle` が**単一の契約の正本**。U3 が出力し U4b がそのまま読みます。**手作業の変換は不要**。ずれた場合は `schema_version` 検証が**既定でエラー終了**させます（黙って誤集計しない） |
| **割当の偏りで特定作品の比較回数が不足し BT 精度が不均一になる** | 割当が露出均衡・層間比率を保証（XC-01）。実測は `/admin/winrates` の `matches` で確認できます。分断は BT 集計が「非連結」warning で明示します |

## 7. データの取り扱い

- **BT 集計は token を一切読みません**（出力にも含みません）。再識別リスクを構造的に排除しています。
- **`export.json` / `bt_result.json` / `tokens.dist.txt` はリポジトリ管理外**＝保全はあなたの責任です。
- **取得と推定は分離**してください。`export.json` に固定してから集計すれば、**同一ファイル → 常に同一結果**（行順序が変わっても不変）。スナップショットの監査単位がファイルで閉じます。
