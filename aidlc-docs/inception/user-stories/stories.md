# ユーザーストーリー (User Stories) — nazokake-judge

**作成日**: 2026-07-12
**分解アプローチ**: Hybrid（参加者 = User Journey / 研究者 = Feature-Based）
**受入基準フォーマット**: 主要フロー = Given-When-Then、静的制約 = チェックリスト（併用）
**粒度**: 中粒度（機能まとまり単位）

構成:
- **A. 参加者ジャーニー**（P-EVAL）: US-P01〜US-P08（フロー順）
- **B. 研究者フィーチャー**（P-RSCH）: US-R01〜US-R06
- **C. 横断制約ストーリー**（NFR/PBT 重点）: XC-01〜XC-04

各ストーリーは INVEST を満たすよう記述。関連要件（FR/NFR）を明示し、PBT 重点箇所は該当ストーリーおよび XC で二重に追跡する。

参加者フロー全体像:
`トークンアクセス → 教示 → 練習試行 → ペア判定 ×N → ブリッジ Likert → 事後アンケート → 完了`（中断・再開はフロー上の任意イベント）

---

## A. 参加者ジャーニー（Persona: P-EVAL）

### US-P01: 個別トークンでセッションにアクセスする
**ストーリー**: 評価者として、配布された個別トークン付き URL でアクセスしたい。なぜなら、アカウント登録の手間なく、自分の回答が正しく紐付く形で評価を始めたいからだ。
**関連要件**: FR-01, NFR-04, NFR-08

**受入基準（Given-When-Then）**
- Given 有効な未使用トークンの URL、When アクセスする、Then セッション開始画面（教示）が表示され、以降の回答がそのトークンに紐付く。
- Given 無効・存在しないトークン、When アクセスする、Then エラーメッセージを表示し、判定画面には進めない。
- Given 既に完了したトークン、When 再アクセスする、Then 「完了済み」である旨を表示する（新規回答は受け付けない）。

**受入基準（チェックリスト）**
- [ ] トークンは推測困難（十分なエントロピー）である。
- [ ] 通信は HTTPS。
- [ ] トークン以外の個人特定情報を要求しない。

---

### US-P02: 教示と練習試行で評価方法を理解する
**ストーリー**: 評価者として、判定を始める前に評価方法の説明と練習試行を行いたい。なぜなら、本番の判定に入る前に操作と評価観点を理解し、迷わず回答したいからだ。
**関連要件**: FR-04（セッション構成）

**受入基準（Given-When-Then）**
- Given セッション開始、When 教示画面を読み進める、Then 練習試行（本番集計に含めないダミーペア）に進める。
- Given 練習試行を完了、When 「本番を開始」する、Then 最初の本番ペアが表示される。

**受入基準（チェックリスト）**
- [ ] 練習試行の回答は本番データ（集計対象）に含めない。
- [ ] 教示は日本語のみ。

---

### US-P03: ペアを比較して優劣を判定・送信する
**ストーリー**: 評価者として、提示された 2 作品（A/B）を読み比べて優れた方を選び送信したい。なぜなら、それが本評価タスクの中核だからだ。
**関連要件**: FR-02, FR-05, NFR-02, NFR-03

**受入基準（Given-When-Then）**
- Given 本番ペアが表示されている、When A または B を選んで送信する、Then その回答が即時にサーバへ保存され、次のペアが表示される。
- Given 回答送信がネットワーク要因で失敗、When 再試行する、Then 二重登録されずに 1 件だけ保存される（同一ペア・同一トークンの回答は冪等に扱う）。

**受入基準（チェックリスト）**
- [ ] スマートフォンで A/B が縦積みでも両作品が読みやすい（モバイルファースト）。
- [ ] 選択前は送信できない（無回答送信の防止）。
- [ ] 回答は 1 ペアごとにサーバ保存される（FR-05）。

---

### US-P04: セッションの進捗と残量を把握する
**ストーリー**: 評価者として、あと何ペア残っているか進捗を把握したい。なぜなら、25〜35 分のタスクの見通しを持ち、最後まで続けやすくしたいからだ。
**関連要件**: FR-04

**受入基準（Given-When-Then）**
- Given 判定中、When 各ペアを回答する、Then 進捗（済み/全体、または残数）が更新表示される。

**受入基準（チェックリスト）**
- [ ] 進捗表示はモバイルでも視認できる位置にある。

---

### US-P05: ブリッジ Likert 項目に回答する
**ストーリー**: 評価者として、指定された一部の作品に Likert 尺度で評定したい。なぜなら、それが BT 尺度を較正するアンカーになるからだ。
**関連要件**: FR-06

**受入基準（Given-When-Then）**
- Given ブリッジ Likert 項目が割り当てられている、When 尺度で評定して送信する、Then 回答がトークンに紐付いて保存され、フローが継続する。

**受入基準（チェックリスト）**
- [ ] Likert 回答は BT 尺度較正用として、ペア判定と区別可能な形で保存される。
- [ ] Likert 項目の具体設計はプール確定後に確定（暫定でストーリー化）。

---

### US-P06: 事後アンケートに回答する
**ストーリー**: 評価者として、判定終了後に事後アンケートへ回答したい。なぜなら、評価者相対性の統制や熟達度に応じた分析のための情報を提供するためだ。
**関連要件**: FR-07

**受入基準（Given-When-Then）**
- Given 全ペア判定と Likert が完了、When 事後アンケートに回答して送信する、Then 回答がトークンに紐付いて保存される。

**受入基準（チェックリスト）**
- [ ] 収集項目: (i) なぞかけ経験・自己申告熟達度、(ii) ドメイン馴染み・経験様態、(iii) 判定時に重視した観点、(iv) 年代程度のデモグラフィック。
- [ ] 設問の最終確定は刺激プール確定後（暫定構成でストーリー化）。

---

### US-P07: セッションを完了する
**ストーリー**: 評価者として、すべて回答し終えたら明確な完了画面を見たい。なぜなら、自分のタスクが正しく終わったことを確認して安心したいからだ。
**関連要件**: FR-04, FR-05

**受入基準（Given-When-Then）**
- Given 事後アンケートまで完了、When 送信する、Then 完了画面が表示され、当該トークンは以後「完了済み」として扱われる。

**受入基準（チェックリスト）**
- [ ] 完了後の再アクセスは US-P01 の「完了済み」挙動に一致する。

---

### US-P08: 中断したセッションを再開する
**ストーリー**: 評価者として、途中で離脱しても同じトークンで続きから再開したい。なぜなら、25〜35 分の間に離脱が現実的に起こり、最初からやり直したくないからだ。
**関連要件**: FR-05, NFR-05, NFR-07（PBT-02 ラウンドトリップ）

**受入基準（Given-When-Then）**
- Given あるトークンで数ペア回答済みの状態で離脱、When 同一トークンで再アクセスする、Then 未回答の次のペアから再開し、既回答は保持される。
- Given 保存済みセッション状態、When 復元する、Then 復元後の状態は保存前と論理的に等価である（ラウンドトリップ）。

**受入基準（チェックリスト）**
- [ ] 再開時に既回答ペアを重複提示しない。
- [ ] セッション状態のシリアライズ/デシリアライズがラウンドトリップ性を満たす（→ XC-02）。

---

## B. 研究者フィーチャー（Persona: P-RSCH）

### US-R01: 実験進捗をモニタリングする
**ストーリー**: 研究者として、参加者数と回答数を確認したい。なぜなら、実験がどこまで進み、十分なデータが集まったか判断したいからだ。
**関連要件**: FR-09, NFR-01

**受入基準（Given-When-Then）**
- Given 管理画面にアクセスする権限、When 進捗を開く、Then 参加者数（トークン発行/開始/完了）と総回答数が表示される。

**受入基準（チェックリスト）**
- [ ] 管理画面へのアクセスは限定される（一般の評価者トークンでは開けない）。

---

### US-R02: 回答データをエクスポートする
**ストーリー**: 研究者として、回答データを CSV/JSON でエクスポートしたい。なぜなら、オフライン BT 集計スクリプトへそのまま投入して推定したいからだ。
**関連要件**: FR-09, FR-10, NFR-08

**受入基準（Given-When-Then）**
- Given 収集済みデータ、When エクスポートを実行する、Then ペア判定・Likert・事後アンケートを含むデータが CSV/JSON で出力される。
- Given エクスポートされたファイル、When BT 集計スクリプト（US-R04）に入力する、Then 変換なしでそのまま読み込める（形式が整合する）。

**受入基準（チェックリスト）**
- [ ] エクスポート形式は US-R04 のスクリプト入力仕様と一致する（この整合を本ストーリーで固定）。
- [ ] トークン単位で回答が紐付く（評価者相対性分析のため）。

---

### US-R03: 暫定勝率テーブルを確認する
**ストーリー**: 研究者として、収集途中でも作品ごとの暫定勝率を軽く確認したい。なぜなら、本格集計の前に傾向や異常（極端な偏り）に気づきたいからだ。
**関連要件**: FR-09, FR-10

**受入基準（Given-When-Then）**
- Given 一定数の回答が蓄積、When 暫定勝率テーブルを開く、Then 作品ごとの対戦数・勝率が一覧表示される。

**受入基準（チェックリスト）**
- [ ] これは簡易表示であり、正式な BT 推定ではない旨が明示される。
- [ ] 実装は軽量（工数を抑える）。

---

### US-R04: オフラインで BT 集計を実行する
**ストーリー**: 研究者として、エクスポートデータから BT 推定を実行したい。なぜなら、「データ回収 → 推定 → 新作の位置確認」という判定装置の一巡を回すためだ。
**関連要件**: FR-10, NFR-07

**受入基準（Given-When-Then）**
- Given US-R02 のエクスポートファイル、When BT 集計スクリプト（Python）を実行する、Then 全作品の BT 尺度上の推定位置（品質スコア）が出力される。
- Given 新作を含むデータ、When 集計する、Then 新作のプロ水準に対する相対位置が確認できる。

**受入基準（チェックリスト）**
- [ ] スクリプトは `scripts/` 配下に置く（アプリ本体とは分離）。
- [ ] 入力形式は US-R02 のエクスポート形式と一致する。
- [ ] ブリッジ Likert を較正アンカーとして解釈に利用できる。

---

### US-R05: 参加者トークンを発行・配布する
**ストーリー**: 研究者として、参加者ごとの個別トークンを発行・配布したい。なぜなら、それが US-P01（評価者のアクセス）と US-R01（発行数モニタリング）の前提であり、これがないと実験を開始できないからだ。
**関連要件**: FR-01, NFR-04, NFR-08

**受入基準（Given-When-Then）**
- Given 発行数を指定、When トークン発行スクリプト（`scripts/`）を実行する、Then 推測困難な一意トークンが指定数だけ生成され、配布可能な URL 一覧が出力される。
- Given 既存の発行済みトークン、When 追加発行する、Then 既存トークンと衝突しない一意なトークンが生成される。

**受入基準（チェックリスト）**
- [ ] スクリプトは `scripts/` 配下に置く。
- [ ] トークンは十分なエントロピーを持つ（XC-03 の推測困難性と整合）。
- [ ] 発行したトークンは US-R01 の「発行数」モニタリングに反映される。

---

### US-R06: 刺激プールを層ラベル付きで投入する
**ストーリー**: 研究者として、4 層構成の作品を層ラベル付きで DB へ投入したい。なぜなら、それが判定対象の母集団であり、かつ層ラベルは XC-01 の層間比率制約の前提だからだ。
**関連要件**: FR-08, FR-03（層間比率）, NFR-08

**受入基準（Given-When-Then）**
- Given 作品本文と層ラベルを含む入力、When 投入スクリプト（`scripts/`）を実行する、Then 各作品が層ラベル付きでプールに登録される。
- Given 層ラベルが欠落した入力、When 投入を実行する、Then エラーとし、ラベルなしの登録を拒否する。

**受入基準（チェックリスト）**
- [ ] 各作品に 4 層（プロ作品 / AI 生成 / 編集・自作 / ルールベース）のいずれかの層ラベルが**必須**である（XC-01 の層間比率制約の前提）。
- [ ] 投入・変更は `scripts/` 経由（管理 UI は設けない。Requirements Q8=B と整合）。
- [ ] 作品本文（刺激データ）はリポジトリ管理外（NFR-08）。

## C. 横断制約ストーリー（NFR / PBT 重点 — Construction で追跡）

> **PBT 番号の出所（検証可能性）**: Requirements Q14 で PBT を **Partial** と確定した。Partial モードで強制されるルールは、AI-DLC の PBT ルール定義 `property-based-testing.md` の **「Partial Enforcement Mode」** に「only rules **PBT-02, PBT-03, PBT-07, PBT-08, PBT-09**」と明記されている固定セットであり、本ワークフローが恣意的に付番したものではない。ユーザー意図（純粋関数の不変条件 = **PBT-03**、シリアライズのラウンドトリップ = **PBT-02**）に、PBT 実施の基盤ルール（ジェネレータ品質 = PBT-07、縮小・再現性 = PBT-08、フレームワーク選定 = PBT-09）が付随する構成。
>
> 以下は各機能ストーリーの受入基準に組み込みつつ、横断制約として別立てで明示し、Construction フェーズでの追跡を確実にする。

### XC-01: ペア割当の露出均衡・層間比率を保証する
**制約**: ペア割当は「各項目の露出回数を均す」「層間ペアを一定割合含める」制約を満たさねばならない。割当の偏りは BT 推定を直接汚染する。**露出均衡は本来、単一セッション内ではなく全セッション横断の性質**である。参加者数が事前に確定しないため、割当関数は現在の露出状態を入力に取る必要がある。
**関連要件**: FR-03, NFR-07（PBT-03 不変条件）
**設計への申し送り（Application Design）**: 割当関数のシグネチャを `(プール, これまでの露出カウント, シード) → ペア列` とし、現在の露出状態を入力に取る純粋関数として設計すること。この入力設計を落とさない。

**受入基準（Given-When-Then）**
- Given プール・これまでの露出カウント・シード、When 次のセッション用ペア列を生成する、Then 割当関数は `(プール, 露出カウント, シード) → ペア列` の純粋関数であり、返されたペアで露出カウントを更新すると全体の露出偏りが許容範囲へ収束する方向に働く。
- Given 多数セッション横断で累積した露出、When 各項目の露出回数を集計する、Then 偏りが許容範囲に収まる（**全体不変条件**）。
- Given 生成された多数のセッション、When 層間ペア比率を集計する、Then 指定割合を満たす。

**受入基準（チェックリスト）**
- [ ] 割当関数は「(プール, これまでの露出カウント, シード) → ペア列」の純粋関数とする（現在の露出状態を入力に取る）。
- [ ] 露出均衡は**全セッション横断**の性質として評価する（単一セッション内に閉じない）。
- [ ] 各作品の層ラベル（US-R06 で投入）を入力に用いる（層間比率制約の前提）。
- [ ] 割当ロジックは純粋関数として実装し、PBT-03（不変条件）で重点検証する。
- [ ] 露出均衡・層間比率の許容範囲を明示的に定義する。

---

### XC-02: セッション状態のシリアライズをラウンドトリップ保証する
**制約**: 中断・再開のためのセッション状態は、シリアライズ→デシリアライズで元と等価でなければならない。
**関連要件**: FR-05, NFR-07（PBT-02 ラウンドトリップ）

**受入基準（Given-When-Then）**
- Given 任意の有効なセッション状態、When シリアライズしてデシリアライズする、Then 元の状態と論理的に等価である。

**受入基準（チェックリスト）**
- [ ] ラウンドトリップを PBT-02 で検証する（生成入力を使用）。

---

### XC-03: 基本セキュリティ衛生を満たす
**制約**: Security 拡張は OFF だが、基本衛生は通常実装として必須。
**関連要件**: NFR-04

**受入基準（チェックリスト）**
- [ ] HTTPS 強制。
- [ ] トークン推測困難性（十分なエントロピー）。
- [ ] SQL インジェクション対策（パラメータ化クエリ等）。
- [ ] 適切な CORS 設定。

---

### XC-04: モバイルファースト・レスポンシブ・日本語 UI
**制約**: PC・スマホ両対応（モバイルファースト）、UI は日本語のみ。
**関連要件**: NFR-02, NFR-03

**受入基準（チェックリスト）**
- [ ] 主要画面（判定・Likert・アンケート）がスマホで破綻なく操作できる。
- [ ] A/B 2 作品が縦積みでも読みやすい（US-P03 と整合）。
- [ ] UI 文言は日本語のみ。

## INVEST 準拠・トレーサビリティ確認

- **Independent**: 参加者ジャーニーは中粒度の機能まとまりで分割。研究者フィーチャーは相互独立（US-R02 と US-R04 のみ入出力形式で意図的に接続）。
- **Negotiable**: Likert 項目・事後アンケート設問は暫定（プール確定後に確定）とし交渉余地を明示。
- **Valuable**: 各ストーリーに「なぜなら」で価値を明記。
- **Estimable / Small**: 中粒度で見積り可能な規模に調整。
- **Testable**: 主要フローを Given-When-Then、静的制約をチェックリストで検証可能に記述。

**ペルソナ→ストーリー**: P-EVAL → US-P01〜08 / P-RSCH → US-R01〜06, XC-01〜04（品質責任）
**装置の一巡（研究者）**: US-R06（プール投入）→ US-R05（トークン発行）→ [参加者ジャーニー] → US-R01（進捗）→ US-R02（エクスポート）→ US-R04（BT 集計）→ 新作の位置確認
**要件カバレッジ**: FR-01〜10, NFR-01〜09 をいずれかのストーリー/横断制約で参照。
