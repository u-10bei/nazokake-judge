# ペルソナ (Personas) — nazokake-judge

**作成日**: 2026-07-12
**分解アプローチ**: Hybrid（参加者 = User Journey / 研究者 = Feature-Based）
**ペルソナ数**: 2（評価者 / 研究者）

> 注: 評価者の経験・熟達度は多様だが、アプリの挙動を変えないため単一ペルソナとして扱い、多様性は下記「特性」で言及する（分析側で事後アンケートの申告値を用いて区別する）。

---

## Persona 1: 評価者（実験参加者）— 「ジャッジ」

| 項目 | 内容 |
|---|---|
| **ID** | P-EVAL |
| **役割** | 提示された 2 作品のなぞかけを比較し、優劣を判定する日本語話者 |
| **アクセス手段** | 研究者から配布された個別トークン付き URL（アカウント登録なし） |
| **利用環境** | スマートフォン利用が中心（モバイルファースト）。PC も可 |
| **セッション** | 1 回 25〜35 分。数十ペアの A/B 判定 + ブリッジ Likert + 事後アンケート |

**特性・背景（多様性）**
- なぞかけ経験は様々（頻繁に触れる経験者から、ほぼ初めての初心者まで）。
- 熟達度・ドメイン馴染みも様々で、これらは事後アンケートで自己申告として記録される。
- 掛詞（日本語）を理解する日本語話者に限定される。

**動機・ゴール**
- 依頼された評価タスクを、迷わず・負担少なく最後まで完了したい。
- 各ペアで「どちらが優れたなぞかけか」を直感的に選びたい。
- 途中で中断しても、後で続きから再開したい。

**不満・障壁（Pain Points）**
- 長いセッションでの離脱（時間切れ・通信断・誤操作）。
- 2 作品が読みにくい / 比較しづらいレイアウト。
- 操作方法や評価基準が不明瞭なまま始まること。

**このペルソナが関わる主なストーリー**: US-P01〜US-P08（参加者ジャーニー全般）

## Persona 2: 研究者（実験運用者 / 管理者）— 「オペレーター」

| 項目 | 内容 |
|---|---|
| **ID** | P-RSCH |
| **役割** | 実験を設計・運用し、回答データを回収して Bradley–Terry 推定を行う研究者 |
| **アクセス手段** | 管理用の限定的な画面 + `scripts/` 配下の CLI ツール |
| **利用環境** | PC。データ分析は Python 環境（オフライン集計） |
| **技能** | 統計・スクリプトに習熟。刺激プールの投入・変更はスクリプトで実施 |

**特性・背景**
- 刺激プールは 4 層構成・計 90〜95 件を管理（投入・変更はスクリプト経由、管理 UI は使わない）。
- 「制作物がプロ水準に対しどこに位置するか」を反復的に判定する装置として本システムを運用する。

**動機・ゴール**
- 実験の進捗（参加者数・回答数）を把握したい。
- 回答データを、オフライン BT 集計スクリプトへそのまま投入できる形式でエクスポートしたい。
- 収集途中でも、暫定的な勝率の傾向を軽く確認したい。
- 「データ回収 → BT 推定 → 新作の位置確認」という装置の一巡を回したい。

**不満・障壁（Pain Points）**
- エクスポート形式と集計スクリプトの入力形式がずれて、手作業の変換が発生すること。
- 割当ロジックの偏りにより特定作品の比較回数が不足し、BT 推定の精度が不均一になること。

**このペルソナが関わる主なストーリー**: US-R01〜US-R06（研究者フィーチャー: 進捗 / エクスポート / 暫定勝率 / BT 集計 / トークン発行 / プール投入）、横断制約 XC-01〜XC-04
