# 要件定義書 (Requirements) — nazokake-judge

**プロジェクト**: nazokake-judge（なぞかけ品質判定 Web アプリ）
**フェーズ**: INCEPTION / Requirements Analysis
**作成日**: 2026-07-12
**深度**: Standard

---

## 1. インテント分析 (Intent Analysis Summary)

| 項目 | 内容 |
|---|---|
| **ユーザー要求** | 「Using AI-DLC, なぞかけ判定 Web アプリの要求定義を始めたい」 |
| **リクエスト種別** | New Project（Greenfield） |
| **スコープ** | System-wide（Web アプリ全体 + オフライン集計スクリプト） |
| **複雑度** | Moderate〜Complex（統計モデル・実験設計・多層刺激プール） |

**目的**: なぞかけの品質を、個人の主観や単発の印象評定ではなく **ペア比較（A/B 判定）にもとづく Bradley–Terry モデル**で安定測定し、制作物がプロ作品水準に対してどこに位置するかを反復判定できる「品質判定装置」を構築する。本アプリは機知（wit）評価研究の**人間評価インフラ**である。

---

## 2. 確定インプット（README 由来・所与の前提）

- **判定方式**: 2 作品を並べて優劣を選ぶペア比較（A/B 判定）
- **統計モデル**: Bradley–Terry モデルで全作品を単一品質尺度上に配置
- **補助指標**: 一部項目に Likert 評定を併用し、BT 尺度上の位置の較正アンカーとする
- **刺激プール**: 出自の異なる 4 層（プロ作品層 / AI 生成層 / 編集・自作層 / ルールベース生成層）
- **セッション構成**: 数十ペアの比較 + 少数のブリッジ Likert 項目 + 事後アンケート（所要 25〜35 分）
- **デプロイ**: 実験用サブドメインに分離。運用基盤（ドメイン・契約）は確保済み
- **データ管理方針**: 刺激データ（作品本文）・参加者回答データはリポジトリに含めない（別経路投入・管理外）

## 3. 機能要件 (Functional Requirements)

### FR-01: 評価者アクセス（個別トークン認証）
- 評価者は**参加者ごとに発行する推測困難な個別トークン付き URL** でアクセスする（アカウント登録なし）。
- トークンにより回答を評価者に紐付ける（評価者相対性の分析、経験・熟達度による感度分析、将来の帰属バイアス実験の基盤共用のため）。
- 根拠: Q1=C。

### FR-02: ペア比較 UI
- 2 作品（A/B）を提示し、評価者が優劣を選択する。
- **モバイルファースト**で設計し、A/B が縦積みレイアウトでも読みやすいことを UI 要件とする。
- 根拠: Q4=C、確定インプット。

### FR-03: 比較ペアの割当（計画的制約付きランダム）
- ペアは**制約付きランダム**で選定する。制約は少なくとも次を含む:
  - 各項目の**露出回数を均す**（比較回数の偏りによる BT 推定精度の不均一を防ぐ）。
  - **層間ペア**（異なる出自層どうしの比較）を一定割合含める。
- セッションごとに抽選する。
- 適応的サンプリングは v1 では採用せず、将来の反復判定フェーズで再検討する。
- 根拠: Q2=X（計画的制約付きランダム）。

### FR-04: セッション進行
- 1 セッション = 数十ペアの比較 + 少数のブリッジ Likert 項目 + 事後アンケート。想定所要 25〜35 分。
- 根拠: 確定インプット。

### FR-05: 回答の逐次保存と中断・再開
- 回答は **1 ペアごとにサーバへ保存**する。
- 同一トークンでの再アクセス時に**続きから再開**できる。
- 根拠: Q6=A。

### FR-06: ブリッジ Likert 評定
- 一部項目に Likert 評定を実施し、BT 尺度の較正アンカーとして解釈可能にする。
- 根拠: 確定インプット。

### FR-07: 事後アンケート
- 次の構成で収集する:
  1. なぞかけ経験の有無・自己申告の熟達度
  2. ドメイン馴染み・経験様態に関する設問（落差の評価者相対性の統制用）
  3. 判定時に重視した観点
  4. デモグラフィックは年代程度に留める
- **設問の最終確定は刺激プール確定後**に行う（後続タスク）。
- 根拠: Q9=X。

### FR-08: 刺激プール
- 初期投入は 4 層構成で計 **90〜95 件程度**（プロ約 30 / AI 生成約 20 / 編集・自作約 30 / ルールベース約 10〜15）。
- 刺激プールの**投入・変更はスクリプト（`scripts/`）経由**で行う。アプリ内の管理 UI は実装しない。
- 根拠: Q8=B、Q10=B。

### FR-09: 管理者（研究者）向け機能
- **データエクスポート**（CSV/JSON ダウンロード）。
- **進捗モニタリング**（参加者数・回答数の確認）。
- **暫定勝率テーブル**程度の簡易表示（本格集計ではない）。
- 根拠: Q7=C、Q8=B。

### FR-10: 集計方式
- 管理画面: 回答数・暫定勝率テーブルの簡易表示のみ（工数を抑える）。
- 本格的な **BT 推定は Python によるオフラインバッチ集計スクリプト**で行う（回収はアプリ、集計はオフライン）。
- 根拠: Q7=C。

## 4. 非機能要件 (Non-Functional Requirements)

### NFR-01: 規模・性能
- 小規模（総参加者数十名、同時アクセス数名程度）。大規模スケール設計は不要。
- 根拠: Q3=A。

### NFR-02: 対応デバイス・レスポンシブ
- PC・スマホ両対応（レスポンシブ）。モバイルファースト。
- 根拠: Q4=C。

### NFR-03: 言語
- UI は日本語のみ（刺激が日本語の掛詞であり、評価者は日本語話者に限定）。
- 根拠: Q5=A。

### NFR-04: セキュリティ（基本衛生・必須）
- AI-DLC の Security 拡張はスキップ（高機密情報を扱わないため）。ただし**基本的なセキュリティ衛生は通常実装として必ず行う**:
  - HTTPS 通信の強制
  - トークンの推測困難性（十分なエントロピー）
  - SQL インジェクション対策（パラメータ化クエリ等）
  - 適切な CORS 設定
- 根拠: Q12=B（補足）。

### NFR-05: データ保全
- 回答の逐次保存（FR-05）が実質的なデータ保全策。専用の高可用性・DR 設計は行わない。
- 根拠: Q6=A、Q13=B。

### NFR-06: 可用性・回復性
- AI-DLC の Resiliency 拡張はスキップ（小規模実験アプリ）。
- 根拠: Q13=B。

### NFR-07: テスト戦略（PBT Partial）
- プロパティベーステストを **Partial モード**で適用（強制対象: PBT-02 ラウンドトリップ / PBT-03 不変条件 / PBT-07 ジェネレータ品質 / PBT-08 縮小・再現性 / PBT-09 フレームワーク選定。他は助言的）。
- **番号の出所（検証可能性）**: この強制対象セットは AI-DLC の PBT ルール定義 `property-based-testing.md`「Partial Enforcement Mode」に「only rules PBT-02, PBT-03, PBT-07, PBT-08, PBT-09」と規定された固定セットに基づく（ワークフローによる恣意的な付番ではない）。ユーザー意図（純粋関数の不変条件=PBT-03、シリアライズのラウンドトリップ=PBT-02）に、PBT 実施の基盤ルール（PBT-07/08/09）が付随する。
- 重点対象:
  - **ペア割当ロジック**（露出均衡の制約 → 不変条件 PBT-03）。バグが BT 推定を直接汚染するため重点的にテストする。
  - **セッション状態のシリアライズ**（ラウンドトリップ PBT-02）。
- PBT フレームワークは NFR Requirements で言語に応じて選定する（後続タスク）。
- 根拠: Q14=B。

### NFR-08: プライバシー・倫理・データ管理
- 匿名トークンと回答データのみを扱う（個人特定情報は最小限）。
- 刺激データ・参加者回答データはリポジトリ管理外（デプロイ時に別経路投入）。
- 根拠: 確定インプット、Q1=C。

### NFR-09: デプロイ
- 実験用サブドメインに分離してデプロイ。運用基盤（ドメイン・契約）は確保済み。
- 根拠: 確定インプット、Q11 補足。

## 7. 主要要件サマリ

- **何を作るか**: なぞかけ品質を BT モデルで測る人間評価 Web アプリ。個別トークンでアクセスする評価者が、モバイルファーストの UI で数十ペアの A/B 判定 + ブリッジ Likert + 事後アンケートを行う。
- **どう測るか**: 露出均衡・層間比率を制約にした制約付きランダムでペアを割り当て、回答をトークン単位で逐次保存。回収データを Python でオフライン BT 集計。
- **規模と品質**: 小規模（数十名）。基本セキュリティ衛生は必須、可用性・回復性の作り込みは不要。ペア割当ロジックとシリアライズは PBT で重点検証。
- **次に決めること**: アーキテクチャ（案 A/B）、アンケート設問確定、PBT フレームワーク。
